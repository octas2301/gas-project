# 2回目メニュー移動で白画面になる場合の原因確認手順（F12）

## 目的

「1回目の移動はできるが、2回目に別メニューをタップすると白画面になる」とき、**2回目のクリックでブラウザがどのURLにアクセスしているか**を確認し、原因を切り分けます。

---

## やること一覧（ここを渡してもらえれば原因を詰められます）

1. **Console** … 赤いエラーの内容（スクショ or 文言）
2. **Network** … 2回目タップで出たリクエストの「URL」と「Status」
3. **スプレッドシート「デバッグログ」** … 再現した時刻に「request view=」が何回あるか

---

## 手順（どこを開いて、何を送るか）

### ステップ1：Console のエラーを取る（最重要）

アプリは **iframe の中** で動いているため、**Console を「iframe 側」に切り替える**必要があります。

#### 「userCodeAppPanel を選択」とは（どこを押すか）

1. **F12** で開発者ツールを開く。
2. いちばん上のタブの並び（Elements・Console・Sources…）のうち、**「Console」** をクリックする。
3. Console の**中身が表示されているエリアの、いちばん上**（フィルターや「Default levels」の左あたり）に、**「top」** と書いてある**▼付きのボタン**があります。  
   → これが「実行コンテキスト」の選択ボタンです。
4. その **「top」▼** を**クリック**すると、下に**一覧**が開きます。
5. 一覧の中に **「userCodeAppPanel」** という行があります（その下に `n-q52uapmvb3yzgzs2zi5k3wywfxjddgcsg...` のような長い英数字が続くことがあります）。
6. その **「userCodeAppPanel」** の行を**クリック**して選ぶ。  
   → これが「userCodeAppPanel を選択した状態」です。選ぶと、Console に表示されるメッセージが「在庫管理アプリの iframe 内の JavaScript」のものだけになります。

**まとめ**：Console タブを開いたら、**「top」と書いてある▼をクリック → 出てきた一覧の「userCodeAppPanel」をクリック**です。

7. **赤いエラー**（「Uncaught …」など）が表示されていれば：
   - その**エラー文をクリックして展開**し、**スクリーンショット**を撮る、または
   - エラー文を**右クリック → コピー**して、メモ帳に貼り付けて保存する。
8. エラーが複数ある場合は、**すべて**の赤いエラーの内容を撮る／コピーする。
9. **ここまでを「①Consoleの内容」として保管** → 後で教えてください。

※「top」のままの場合、親ページのエラーも混ざります。可能なら iframe（userCodeAppPanel）を選ぶと、在庫管理アプリ専用のエラーだけ見られます。

---

### ステップ2：Network で「2回目タップ」のリクエストを確認する

1. F12 の **「Network」（ネットワーク）** タブをクリックする。
2. **「Disable cache」** にチェックを入れる。
3. 一覧が多ければ、フィルタ欄に **`exec`** と入力して絞る。
4. **再現する**：
   - いったんアプリの画面を出荷に戻す（必要ならページを再読み込み）。
   - **1回目**：メニューで「移動」をタップ → 移動画面になることを確認。
   - **2回目**：メニューで「入荷」をタップ → 白画面になる。
5. Network 一覧で、**2回目のタップの直後に増えた行**を探す。
   - **Name** に `exec` や `userCodeAppPanel`、**URL** に `view=` が出ている行をクリックする。
6. 右側（または下）の **Headers** で **「Request URL」** の全文を確認する。
   - その **URL 全文** をコピーするか、**スクリーンショット**を撮る。
7. 同じリクエストの **Status**（200, 304, 404 など）をメモする。
8. **「Response」タブ**を開き、中身が「HTML がずらっと並んでいる」「ほぼ空」「エラー文だけ」のどれかメモする。
9. **ここまでを「②NetworkのURL・Status・Responseの様子」として保管** → 後で教えてください。

---

### ステップ3：デバッグログで「request view=」の回数を確認する

1. 在庫管理で使っている **Google スプレッドシート** を開く。
2. 一番下のシートで **「デバッグログ」** を開く。
3. ステップ2で再現した **日時** のあたりの行を見る。
4. **「request view=」** という文字が、その時刻あたりに **1回だけか・2回あるか** を数える。
   - 例：1回目タップで `request view=move`、2回目タップで `request view=receiving` のように **2回** 出ていれば「2回」。
5. **「③デバッグログに request view= が 1回か 2回か」** をメモ → 教えてください。

---

## 渡してもらえると助かるもの（まとめ）

| 番号 | 内容 |
|------|------|
| ① | **Console** の赤いエラー（iframe を選んだ状態が理想）の全文 or スクショ |
| ② | **Network** の、2回目タップのリクエストの「URL」「Status」「Response の様子」 |
| ③ | **デバッグログ** に「request view=」が 1回か 2回か |

①②③のうち、できる範囲で教えていただければ、次の対策を具体的に提案できます。

---

## 以下は詳細な手順（必要なら参照）

### 1. 事前準備

1. 在庫管理アプリのURLを開く（出荷画面が表示される状態にする）。
2. **F12** を押して開発者ツールを開く。
3. **Network（ネットワーク）** タブを開く。
4. **「Disable cache」** にチェックを入れておく（キャッシュを無効化）。
5. 一覧が多くなりすぎる場合は、フィルタ欄に **`exec`** または **`script.google`** と入力して、GAS 関連の通信だけに絞る。

### 2. 再現操作

1. **1回目**: 下部メニューで「移動」や「入荷」など、**出荷以外**をタップする。
   - 画面が切り替われば成功。
2. **2回目**: もう一度、別のメニュー（例: 「入荷」→「棚卸」）をタップする。
   - ここで白画面になる想定。

### 3. 確認すること（Network タブ）

2回目のタップの**前後**で、一覧に **新しく1件** リクエストが増えているはずです。

- **2回目のタップに対応するリクエストを探す**
  - 時刻で並べ替えている場合は、いちばん新しい行。
  - **Name** や **URL** が `exec` や `view=` を含む行をクリックする。

確認する項目:

| 項目 | どこで見る | 正常な例 | 要チェックな例 |
|------|------------|----------|----------------|
| **リクエストURL** | クリックした行の「Headers」や一覧のURL列 | `https://script.google.com/.../exec?view=receiving` のように **view=○○** が付いている | `script.google.com` ではなく別ドメイン（例: `googleusercontent.com` の別パス）になっている、または **view=** が付いていない |
| **Status** | 一覧の Status 列 | 200 | 304, 404, 500 など |
| **Response（レスポンス本体）** | そのリクエストを選択 → **Response** または **Preview** タブ | 在庫管理のHTML（出荷・移動などの画面が書かれたHTML）が返っている | 空、または「このアプリは～」以外の短いエラーHTMLだけ |

### 4. 切り分けの目安

- **2回目のタップで、exec?view=○○ へのリクエストが一覧にない**
  - ブラウザが別のURLに遷移している可能性が高いです。
  - 画面上部の「表示: ○○ | base: あり/なし」や、**Address bar（アドレスバー）** に表示されているURLが、2回目タップ前後でどう変わるかも確認してください。
- **exec?view=○○ へのリクエストはあるが、Status が 200 ではない**
  - その Status の数字（404, 500 など）と、可能なら **Response** の内容を控えてください。
- **exec?view=○○ へのリクエストがあり、Status 200 だが Response が空や短い**
  - GAS 側でエラーになっているか、別のデプロイが動いている可能性があります。スプレッドシートの「デバッグログ」シートに、同じ時刻に **request view=○○** が記録されているか確認してください。

### 5. コンソール（Console）タブ

- 2回目タップで白画面になった直後に **Console** タブを開く。
- **赤いエラー** が出ていないか確認する。
- 出ている場合は、**エラー文**と**ファイル名・行番号**をメモする（スクリーンショットでも可）。

### 6. サーバー側の記録（デバッグログ シート）

- スプレッドシートの **「デバッグログ」** シートを開く。
- 再現操作（1回目・2回目のタップ）をした時刻あたりの行を確認する。
- **「request view=move」「request view=receiving」** のように、**2回分**の doGet が記録されているか見る。
  - 2回分あれば、2回目のタップは GAS まで届いているが、返却されたHTMLが表示されていない可能性（iframe や CSP など）。
  - 1回分しかない場合は、2回目のタップでは GAS が呼ばれていない（別URLへ遷移している）可能性が高いです。

---

## 報告してもらえると助かる情報

原因を詰めるため、次のどれか（できる範囲で）を教えてください。

1. **2回目のタップ時、Network タブで増えたリクエストの「URL」**（全文またはスクショ）
2. そのリクエストの **Status** と、**Response タブに表示されている内容**（空か、HTMLの一部か、エラー文か）
3. **デバッグログ** に、再現した時刻に **request view=** が何回出ているか（1回だけか、2回あるか）
4. 白画面のとき **Console** に表示されているエラー（あればその文やスクショ）

---

## 画面上の診断表示

アプリでは、各画面のいちばん上に次の1行が出ます。

- **表示: shipping** （いま表示している画面）
- **base: あり** または **base: なし** （サーバーが base URL を渡せているか）

2回目に白画面になったとき、**その白い画面にこの1行すら出ていない**場合は、**別のURLのページ**が表示されている（GAS の HTML が届いていない）と判断できます。
