# 広告運用 現状整理と過不足

## 参照ルール
- `.cursor/rules/coding-rule.mdc` に準拠

---

## 1. 運用イメージ（ご共有いただいた内容）

| ステップ | 内容 | 備考 |
|----------|------|------|
| ① | 2週間に1度、スプレッドシートで広告結果を取り込む | **ただし** Amazon・楽天・Yahoo! は **実装未**。広告効果データは**特定フォルダ**に保存している（後述）。 |
| ② | 広告結果から除外ワード・除外商品を選定 | 手動想定 |
| ③ | 除外するものにレ点を付けて各モールへアップロード | 手動 or 未実装 |
| ④ | Amazon：オートで検索したワードのうち、特定条件以上の効果があるものをマニュアルターゲットへ移動 | 未実装 |

## 2. 運用マニュアルCSVの内容（参考：広告運用スプシ）

### 2.1 Amazon 元データ 項目解説（目標・アクション）

| 項目名 | 意味 | 目標数値 | 取るべきアクション |
|--------|------|----------|---------------------|
| 検索用語 | 顧客が実際に検索したキーワード | - | 分析のメイン対象 |
| インプレッション | 表示回数 | 要改善なら単価UP | 露出不足の解消 |
| クリック数 | 流入数 | **15回売上0で除外** | 無駄打ちのカット |
| 売上 | 広告経由の注文金額合計 | - | 収益性の確認 |
| 合計費用 | 広告コスト総額 | **売上の20%以内** | 予算オーバー抑制 |
| CTR | クリック率 | 0.4%以上 | 画像・訴求内容の修正 |
| クリックによる購入率 | CVR | 5%以上 | 商品ページ/LPの改善 |
| **ROAS** | 投資効率 | **200%以上** | 不調なら除外 / 好調なら強化 |

### 2.2 楽天 元データ 項目解説（目標・アクション）

| 項目名 | 意味 | 目標数値 | 取るべきアクション |
|--------|------|----------|---------------------|
| 商品管理番号 | 楽天の商品識別ID | - | 分析・バルク適用のキー |
| CTR(%) | クリック率 | 1.0%以上 | サムネイル画像の修正 |
| クリック数(合計) | 全期間の流入数合計 | **15回売上0で除外** | 無駄打ちのカット |
| 実績額(合計) | 広告費用の実支出額 | **売上の10%以内** | コスト管理 |
| 売上金額(合計720時間) | 直近30日間の売上合計 | - | 収益性のメイン指標 |
| CVR(合計720時間)(%) | 直近30日間の成約率 | 5%以上 | 商品ページ改善 |
| **ROAS(合計720時間)(%)** | 直近30日間の投資効率 | **1000%以上** | 不調で削除 / 好調で強化 |

### 2.3 Yahoo! 元データ
- 上記CSVファイル内には **Yahoo! の項目解説は含まれていません**。
- **方針**：Yahoo! 広告はまだ構築していないため、**楽天の運用に準じる**形で進める（項目・目標値は楽天をベースに設計）。

## 3. 既存GAS（コード.js / Yahoo.js）で実装されている範囲

| 領域 | 実装内容 | 広告運用との関係 |
|------|----------|-------------------|
| 楽天 | 商品CSV出力（normal-item.csv）→ MAKE経由FTP、楽天削除（レ点→item-delete.csv→MAKE）、反映確認・完了メール | **広告レポート取り込み・除外アップロードは未実装** |
| Yahoo! | 出品・在庫更新・削除（「削除CK」レ点→API削除） | **広告レポート取り込み・除外アップロードは未実装** |
| Amazon | 競合価格の取得・反映、リサーチCSVのマスタ反映 | **広告レポート取り込み・オート→マニュアルターゲットは未実装** |
| 共通 | レ点（チェック）による一括操作のパターンは「楽天削除CK」「削除CK」「出品CK」で確立済み | 除外キーワード/除外商品の「レ点→アップロード」に流用可能 |

**結論**: 広告運用の①取り込み・③除外アップロード・④Amazonオート→マニュアルは、現状GASには **一切実装されていない**。

## 4. 過不足の整理

### 4.1 不足している機能（これから実装するもの）

| No | 機能 | プラットフォーム | 優先度の目安 |
|----|------|------------------|--------------|
| 1 | **広告結果の取り込み**（レポート取得→スプレッドシートに反映） | Amazon / 楽天 / Yahoo! | 高（①の基盤） |
| 2 | **除外キーワード・除外商品のレ点→各モールへアップロード** | Amazon / 楽天 / Yahoo! | 高（③） |
| 3 | **Amazon：オートの検索ワードで条件を満たすものをマニュアルターゲットへ移動** | Amazon のみ | 中（④） |

### 4.2 既に揃っているもの（流用・拡張のベース）

- スプレッドシートベースの運用フロー
- 運用マニュアル（目標数値・取るべきアクション）の定義（Amazon・楽天）
- レ点による一括対象選択 → 確認ダイアログ → ファイル/API送信 → レ点解除のパターン（楽天削除・Yahoo削除で実績あり）
- MAKE連携（Drive保存→Webhook送信）の実績（楽天FTP用）

### 4.3 確認済み・方針

| 項目 | 内容 |
|------|------|
| **Yahoo! 広告** | 未構築。**楽天の運用に準じる**形で進める。 |
| **① 広告結果の取り込み** | 広告効果データは**特定フォルダ**に保存している。GAS確認の結果、**広告データ読み取り用のフォルダID・取り込み処理は現行コードに未実装**。 |
| **作業手順** | 広告運用スプシの **「📌運用マニュアル」** と **「📌アップロード手順マニュアル」** に記載あり（当該シートのCSV or スプシIDがあると設計しやすい）。 |
| **③ 各モールへアップロード** | **現状はCSVアップロード**。API連携できるならそれをベストとする。 |
| **優先順位** | **① 広告結果の取り込み** から実装。現状仕様はGASを確認する。 |

## 5. GAS確認結果（広告まわり）

- **コード.js / Yahoo.js** に定義されているフォルダIDは以下のみ（広告専用はなし）。
  - `CSV_SAVE_FOLDER_ID` / `CSV_ARCHIVE_FOLDER_ID`（楽天CSV保存・アーカイブ）
  - `RAKUTEN_GENRE_FOLDER_ID` / `YAHOO_CATEGORY_FOLDER_ID` / `YAHOO_BRAND_FOLDER_ID`（ジャンル・カテゴリ検索用）
  - `DRIVE_IMAGE_SOURCE_FOLDER_ID`（画像未整理）
  - `YAHOO_SUBMISSION_FOLDER_ID`（Yahoo.js：出品用）
- **広告結果の取り込み**（特定フォルダからCSVを読んでスプシに反映する処理）は **未実装**。
- ①を実装するには、**広告結果CSVを保存している「特定フォルダ」のDriveフォルダID** をGAS側で参照する必要がある（設定シート or Script Properties で管理する想定）。

## 6. 次のステップ提案

1. **① 広告結果の取り込み** の要件定義のため、以下を共有いただけると進めやすいです。
   - **広告結果CSVを保存しているGoogleドライブのフォルダID**（またはフォルダ名と「どのドライブか」）。
   - （任意）**「📌運用マニュアル」「📌アップロード手順マニュアル」** の内容（スプシの該当シートをCSV出力してプロジェクトの `参考：広告運用スプシ` に追加、またはスプレッドシートIDを教えていただく）。
2. 上記を踏まえ、**①の要件定義とシステム構成案**（フォルダ監視 or 手動実行、CSV形式の想定、スプシのシート・列構成）を具体的に作成する。
3. Yahoo! は楽天に準じる形で項目・目標値を設計する。③は現状CSVアップロード、可能ならAPI連携を検討する。

## 7. 補足：技術的なポイント（実装時に考慮する点）

- **API利用制限**：各社のレートリミット・1日あたりのクォータを確認し、2週間に1回の一括取得でも遅延・リトライを考慮する。
- **認証**：Amazon Advertising API / 楽天広告API / Yahoo!広告API の認証情報は Script Properties または設定シートで管理し、コードには直書きしない。
- **既存スプレッドシート**：シート名・列名は現状の広告運用スプシに合わせ、既存の「レ点」パターンと整合させる。

以上が現状の整理と過不足です。
