# 在庫管理アプリ 引き継ぎ仕様書

## 0. AppSheet 仕様の引き継ぎ方法（推奨）

### 開発データを「取り出せる」か

- **AppSheet から GAS へそのまま取り出せる開発データはありません。**
  - AppSheet の設定（JSON）のエクスポートはありますが、AppSheet 専用フォーマットのため、GAS にそのままインポートして使うことはできません。
  - 列定義・Ref・数式（App Formula 等）は、**仕様書＋手作業**で GAS 用に書き起こす必要があります。

### 推奨：仕様画像＋この仕様書で引き継ぐ

| 手段 | 内容 | 役割 |
|------|------|------|
| **仕様画像の提供** | Data タブで各テーブルを選択した状態のスクリーンショット（列一覧・型・Ref が写っているもの） | 列名・型・参照先の正確な確認・抜け漏れ防止 |
| **この仕様書** | 下記「2.2 データテーブル別 列名一覧」に、画像と会話履歴から整理した列名・型・Ref を一覧化 | GAS 実装時の唯一の仕様ソースとして使用 |
| **会話履歴の数式** | Initial Value・App Formula・Valid If などの式 | ビジネスロジック（自動計算・バリデーション）の再現用 |

**進め方の提案**

1. **列名一覧の確定**  
   各テーブルについて、Data タブで「Columns」を開いたスクリーンショットを共有いただく。  
   こちらで「2.2 データテーブル別 列名一覧」に反映し、スプレッドシートの列名・AppSheet の型・Ref を確定する。
2. **不足分の補足**  
   特定の列の Initial Value / App Formula など、画像に写っていない設定は、必要に応じて「列名と式」をテキストで共有いただく。
3. **GAS 詳細設計**  
   上記が固まったら、フォルダ構成・doGet/doPost・スマホ UI ワイヤーを記載した「GAS 移行詳細設計」で実現性を確認する。

※ 画像は「在庫移動リスト」分をいただいていれば、そのテーブルは列一覧に反映済みです。他テーブルも同様の形式でいただけると一覧を完成できます。

## 1. 開発の経緯（Gemini との会話から要約）

### 1.1 目的・起点
- **目的**: EC販売のための在庫管理。当初は「スマホでJANスキャン → 数量入力 → スプレッドシート/CSV出力」の**棚卸ツール**として開始。
- **条件**: 永久無料・スマホ対応。**Google AppSheet + Googleスプレッドシート**で実現。

### 1.2 機能の拡張履歴（時系列）

| 段階 | 内容 |
|------|------|
| 初期 | 棚卸スキャンリスト（ID, JAN, 数量, 日時, メモ）。JANに Scan? 有効。 |
| 場所 | メモ → ロケーション（Enum: オフィス, ガレージ, ナーシング, 倉庫豊明, その他）。AllowOtherValues + Input Mode: Dropdown で手入力も可。 |
| マスタ連携 | 商品マスタ（JAN, 商品名, 商品原価）を追加。棚卸のJANを Ref(商品マスタ) にし、スキャン時に商品名・原価を表示。マスタにない場合は「New」でその場登録。 |
| シート名変更 | 最初のシート名を「棚卸スキャンリスト」に変更。接続切れ時は Storage の Worksheet Name を更新 + Regenerate Structure。 |
| 必須・バリデーション | 数量・ロケーションを必須に。Valid If + Invalid value error で日本語メッセージ。数量は Initial Value 0 対策で `[_THIS] <> 0` も併用。 |
| 商品名・原価の保存 | バーチャルではなく実列（商品名・原価）をスプレッドシートに追加し、App Formula で `[JANコード].[商品名]` / `[JANコード].[商品原価]` を保存。 |
| 列順 | ID, JANコード, 商品名, 原価, 数量, 日時, ロケーション。数量は箱入数×箱数+端数で自動計算（箱入数・箱数・端数は Number、Initial Value 0）。 |
| 担当者 | User Settings の Option 1 を「現在の担当者」にし、棚卸の Initial Value に `UserSettings("現在の担当者")`。端末ごとに別名が記録される。 |
| 総原価 | 数量×原価を実列「総原価」に App Formula で保存。Show? OFF でスマホ非表示可。 |
| 連続スキャン | Advance forms automatically ON。出荷は数量=1固定・Auto Save + Auto re-open で「ピッピッ」連続スキャン。 |
| 年月・ピボット | 列「年月」Type=Text, App Formula `TEXT([日時], "yyyy/mm")`。Security filter で今月のみアプリ表示。 |
| 入荷・出荷・移動 | 入荷リスト・出荷リスト・在庫移動リストを追加。出荷は出荷元(Ref: 場所マスタ)。在庫移動は移動元(From)・移動先(To)。場所マスタを別シートで管理。 |
| 差異ダッシュボード | 商品マスタに Virtual Column: 理論在庫(入荷-出荷)、棚卸実数(今月棚卸合計)、在庫差異。Slice「在庫差異あり」で差異≠0のみ表示。Format Rule でマイナスを赤表示。 |
| 箱入数マスタ | 商品マスタに箱入数。入荷/棚卸の箱入数は Initial Value に `[JANコード].[箱入数]`（手入力で上書き可）。 |
| 自動メール | Automation: 毎日指定時刻に「在庫移動リスト」の当日分を表形式でメール。テーブル名は「在庫移動リスト」、列は移動日時・JAN・商品名・移動数量・移動元(From)・移動先(To)・担当者名。無料プランは作成者宛のみ。 |
| 入荷理由 | 理由マスタ（仕入・返品・その他）を作成。入荷リストの入荷理由を Ref(理由マスタ)。 |
| 商品名検索 | 商品マスタに Virtual Column: Google/Amazon 検索リンク。Type=Show, Category=Url, Content に HYPERLINK。App Formula は "" でダミー。Column order で商品名の下に配置。 |

## 3. アプリの骨子（画面・データ流れ）

### 3.1 メニュー（Views）
- **棚卸** … 棚卸スキャンリスト（今月のみ Security filter）
- **入荷** … 入荷リスト
- **出荷** … 出荷リスト（連続スキャン・1点ずつ）
- **在庫移動** … 在庫移動リスト（移動元・移動先選択）
- **商品情報** … 商品マスタ
- **差異確認** … Slice「在庫差異あり」（理論在庫 vs 棚卸実数の差異）

### 3.2 計算ロジック（商品マスタの Virtual Column）
- **理論在庫**: `SUM(入荷リスト[数量]) - SUM(出荷リスト[数量])`（JAN単位・全社合計の簡易版）
- **棚卸実数**: 棚卸スキャンリストの「今月」の数量合計（年月 = TEXT(TODAY(),"yyyy/mm")）
- **在庫差異**: 棚卸実数 - 理論在庫

※ 場所別・出荷元別の理論在庫は、出荷リストの「出荷元」や在庫移動を考慮した式に拡張可能。

### 3.3 運用上の注意（会話から）
- 列追加・削除後は必ず **Regenerate Structure**。
- 見出しの重複・空欄列があると Regenerate が失敗することがある。
- テーブル名は AppSheet 上で「在庫移動リスト」など、シート名と一致しているとは限らないので式では実際のテーブル名を使用。

## 5. 引き継ぎ後の開発で参照すべき点

- **既存のデータ構造を変える場合**: 既存シートの列名・テーブル名をそのまま式やGASで参照するため、変更時は一覧をこのドキュメントに追記すること。
- **AppSheet の制約**: 無料プランではメール送信は作成者宛のみ。複数宛先はGmail転送ルールで対応済みの想定。
- **技術スタックのバージョン**: コーディングルールに従い、明示的に指示されない限りAPI・フレームワークのバージョンは変更しない。

## 7. 棚卸スタート・完了（GAS 実装済み）

### 7.1 目的・概要

- 棚卸に「スタート」「完了」を導入し、棚卸期間中は**入出荷を加味した理論在庫**で棚卸スキャンを突合する。
- 未完了セッションがある間は新規スタートを禁止し、再読み込み時は未完了セッションに復帰して保存・完了できるようにする。

### 7.2 追加シート・列

| 対象 | 内容 |
|------|------|
| **棚卸セッション**（新規シート） | セッションID(UUID), 開始日時(Date), 完了日時(Date/null)。スタートで1行追加、完了で完了日時を記録。 |
| **棚卸スキャンリスト** | 列「セッションID」（文字列・任意）を追加。保存時に sessionId を渡すとそのセッションに紐づく。 |

### 7.3 主要API・戻り値

| API | 説明・戻り値 |
|-----|--------------|
| apiCountStart() | 未完了セッションが1件でもあれば新規作成せず `{ ok: false, error: "未完了の棚卸があります。..." }` を返す。なければ createCountSession() の結果を返す。 |
| apiGetCountSession() | 未完了セッションを1件返す。**返却は Date を含めず** `{ sessionId: string, startTime: string }`（startTime は "yyyy-MM-dd'T'HH:mm:ss"）とする。GAS→クライアントのシリアライズで sessionId が欠落しないよう文字列のみで返す。 |
| apiCountComplete(sessionId) | 指定セッションの完了日時を記録。 |
| apiCount(..., sessionId) | 棚卸1件保存。sessionId を渡すとそのセッションに紐づく。 |
| apiGetVarianceList() | 未完了セッションがあれば `{ list, sessionActive: true }`（list は期間中の理論で突合した差異）。なければ `{ list, sessionActive: false }`（従来の今月棚卸－現在庫）。 |

### 7.4 参照ドキュメント

- 現在庫・差異の計算式、棚卸期間中/期間外、パターン①〜③（入出荷を加味した理論で突合）: `在庫管理アプリ_現在庫と差異の計算ロジック.md` のセクション7。
- 調査用ログ: クライアントは countLog で Console に `[Count]` 付きで出力。サーバーは apiCountStart / apiGetCountSession の結果を debugLogToSheet で「デバッグログ」シートに記録。
