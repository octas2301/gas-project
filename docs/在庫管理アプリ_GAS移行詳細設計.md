# 在庫管理アプリ GAS 移行 詳細設計

## 1. 実現性の確認（結論）

| 項目 | 実現可否 | 条件・リスク |
|------|----------|--------------|
| 同一スプレッドシートを GAS から読書きする | ✅ 可能 | スプレッドシート ID を GAS のプロパティまたは定数で保持。既存 AppSheet 用シートをそのまま使うか、移行後に専用シートに切るかは運用次第。 |
| doGet でスマホ向け HTML を返す | ✅ 可能 | HTML Service の制限（実行時間 30 秒等）内でレンダリング。 |
| 出荷・入荷・棚卸・在庫移動・商品・差異の各機能を GAS で再現する | ✅ 可能 | 各操作を doPost または doGet のクエリで受け、スプレッドシートに追記・参照。Ref に相当する「マスタ参照」は GAS 側で VLOOKUP またはシート読み込みで実装。 |
| スマホで「アプリのように」使える UI | ✅ 可能 | レスポンシブ HTML + メニュー（タブまたはドロップダウン）。ホーム画面に「ウェブアプリ」として追加すればアプリ風に起動可能。 |
| バーコードスキャン | △ 要工夫 | 端末の「バーコード入力」対応キーボード／外部スキャナなら input にフォーカスで入力可能。カメラで読み取る場合は getUserMedia + クライアント側 JS（または PWA）が必要で、GAS 単体では難しいため別途検討。 |
| 自動メール（在庫移動日報） | ✅ 可能 | GAS の Time-driven トリガーで日次実行。MailApp で HTML テーブルを送信。 |
| 同時利用（AppSheet と GAS） | △ 移行期間のみ | 同じシートを両方から触る場合は、列の追加・削除を仕様書で管理し、競合を避ける。本番はどちらか一方に統一する想定。 |

**総合判断**: **GAS への移行は実現可能です。** 大きな変更にはなりますが、フォルダ分離・段階移行・既存データの流用でリスクを抑えられます。

---

## 2. フォルダ構成（推奨）

gas-project は複数 GAS プロジェクトを格納する親フォルダのため、**在庫管理アプリはサブフォルダで独立した scriptId を持つ**形にします（`.cursor/rules/gas-multi-project.mdc` に準拠）。

```
gas-project/
├── .clasp.json              ← 親（既存・Yahoo/コード.js 用）。skipSubdirectories: true
├── コード.js
├── Yahoo.js
│
└── inventory-app/           ← 在庫管理アプリ専用（新規）
    ├── .clasp.json          ← 在庫用 scriptId（別プロジェクト）
    ├── appsscript.json
    ├── Code.gs              ← エントリ・共通（doGet, doPost, スプレッドシート取得）
    ├── SheetService.gs      ← 各シートの読書き・Ref 解決
    ├── InventoryLogic.gs    ← 理論在庫・棚卸実数・差異・平均原価などの計算
    ├── MailJob.gs           ← 在庫移動日報メール（トリガー登録用）
    ├── index.html           ← スマホ向け SPA の土台（メニュー＋コンテンツエリア）
    ├── css/
    │   └── app.css          ← レスポンシブ用スタイル
    └── js/
        └── app.js           ← 画面切替・フォーム送信・一覧表示
```

- **push 時**: `cd gas-project/inventory-app` で `clasp push` を実行する。
- **スプレッドシート**: 既存の「在庫管理アプリ用」1 ファイルを参照する想定。ID は Script Properties または `appsscript.json` の定数で保持。

## 3. doGet / doPost の役割と設計

### 3.1 基本方針

- **doGet**: 常に同じ `index.html` を返す。クエリパラメータで「初期表示する画面」を指定可能（例: `?view=shipping`）。
- **doPost**: フォーム送信や一覧取得の API として使用。パラメータで「どの機能か」と「データ」を受け取り、スプレッドシートに書き込みまたは読み取り結果を JSON で返す。
- 認証: デプロイを「自分（または組織）のみ」にすれば、Google ログイン済みユーザーのみアクセス可能。Session で「担当者名」を保持する場合は、ログイン後に設定画面で 1 回入力して保存する想定。

### 3.2 doGet の役割

| 役割 | 説明 |
|------|------|
| HTML 配信 | `return HtmlService.createHtmlOutputFromFile('index').setXFrameOptionsMode(HtmlService.XFrameOptionsMode.ALLOWALL)` で index.html を返す。 |
| 初期画面 | クエリ `view` が無ければ「メニュー」、`view=shipping` なら出荷、`view=receiving` なら入荷、など。 |
| 静的アセット | CSS/JS は index.html から同じプロジェクトの HTML ファイルとして読み込むか、インラインで記述。 |

```text
例: https://script.google.com/.../exec           → メニュー
    https://script.google.com/.../exec?view=shipping  → 出荷画面
```

### 3.3 doPost の役割（機能別）

| 機能 | 想定パラメータ例 | 処理内容 | 戻り |
|------|------------------|----------|------|
| 出荷登録 | action=shipping, jan, locationFrom, ... | 出荷リストシートに 1 行追加（数量=1） | 成功/失敗 |
| 入荷登録 | action=receiving, jan, qty, unitCost, reason, ... | 入荷リストに 1 行追加。必要なら平均原価を商品マスタに反映 | 成功/失敗 |
| 棚卸登録 | action=count, jan, qty, location, ... | 棚卸スキャンリストに 1 行追加 | 成功/失敗 |
| 在庫移動登録 | action=move, jan, qty, from, to, ... | 在庫移動リストに 1 行追加 | 成功/失敗 |
| 商品マスタ取得 | action=productList または productLookup, jan=... | 商品マスタを検索して返す（Ref の代わり） | JSON |
| 一覧取得 | action=list, table=inventory_move&yyyy mm=... | 指定テーブルの行を返す（差異一覧など） | JSON |
| 設定保存 | action=setUser, currentUser=... | ユーザー設定を PropertiesService 等に保存 | 成功/失敗 |

- 送信方法: フォームから `google.script.run` で GAS の関数を呼ぶ形でも、`fetch(url, { method: 'POST', body: formData })` で doPost を叩く形でも可。どちらも実現可能。
- Ref の扱い: 「JANコード」→ 商品マスタの行を取得して「商品名」「原価」を埋める、「移動元」→ 場所マスタの Key に該当する値で保存、などはすべて GAS 側（SheetService / Code.gs）で実装。

## 4. スマホ UI ワイヤー（ワイヤーフレーム）

### 4.1 全体構成

- **1 ページ構成（SPA 風）**: 上部にタイトル、その下に「メニュー」、メインエリアで出荷・入荷・棚卸・在庫移動・商品情報・現在庫・差異のいずれかを表示。
- メニューは常に表示（固定フッターまたはハンバーガー）し、タップでメインエリアだけ切り替え。URL は `#shipping` などハッシュで揃え、doGet の `view` と連動させてもよい。

### 4.2 メニュー（ナビゲーション）

画像の AppSheet 下部ナビに相当する 6 項目を、横スクロールまたは 2 行で並べる。

| 表示名 | 役割 | 対応 view |
|--------|------|-----------|
| 出荷 | 出荷登録（連続スキャン想定） | shipping |
| 在庫移動 | 移動元・移動先を指定して登録 | move |
| 入荷 | 入荷登録（数量・理由等） | receiving |
| 棚卸 | 棚卸スキャン登録 | count |
| 差異 | 理論在庫 vs 棚卸の差異一覧 | variance |
| 商品情報 | 商品マスタの参照・編集 | products |

※ 「現在庫」は「差異」画面に理論在庫を併記するか、別タブ「現在庫」として追加可能。

### 4.3 各画面のワイヤー（テキスト）

**出荷（shipping）**

- 入力: JAN（バーコード入力欄 1 つ。連続スキャン時は送信後にフォーカスを戻す）、出荷元（場所マスタから選択）。
- 送信で 1 件追加。成功時は同じ画面のまま JAN を空にして次のスキャン待ち。

**在庫移動（move）**

- 入力: JAN、商品名（参照表示）、移動数量、移動元（From）、移動先（To）、担当者名（設定から引き継ぎ可）。
- 送信で在庫移動リストに 1 行追加。

**入荷（receiving）**

- 入力: JAN、商品名・原価（参照表示）、箱入数・箱数・端数（または数量直接）、入荷理由、担当者。
- 送信で入荷リストに 1 行追加。平均原価更新は GAS 側で実施。

**棚卸（count）**

- 入力: JAN、商品名・原価（参照）、箱入数・箱数・端数、数量（計算表示）、ロケーション、担当者。
- 送信で棚卸スキャンリストに 1 行追加。今月のみ表示するフィルタは一覧取得時に GAS で実施。

**差異（variance）**

- 表示: 商品マスタベースで「理論在庫・棚卸実数・在庫差異」を一覧。差異 ≠ 0 のみ表示するオプション。
- 商品名・理論在庫・棚卸実数・差異の列。差異が負の場合は赤表示（CSS）。

**商品情報（products）**

- 一覧: 商品マスタの JAN、商品名、原価、箱入数 等を表表示。
- 新規追加・編集: JAN、商品名、原価、箱入数 を入力して保存。JAN で検索リンク（Google/Amazon）は a タグで同じ式を再現可能。

### 4.4 共通要素

- **担当者**: 画面上部または設定モーダルで「現在の担当者」を 1 回入力し、Session または PropertiesService に保存。各フォームの初期値として使用。
- **バーコード**: まずは `<input type="text">` にフォーカスし、端末のバーコードキーボード／外付けスキャナで入力。カメラ読み取りは将来拡張とする。

## 5. 実装の進め方（推奨順）

1. **inventory-app の作成**: `clasp create` で新規プロジェクトを作り、上記フォルダ構成で `Code.gs` と `index.html` の最小版（doGet で HTML 表示のみ）をデプロイし、スマホで開けることを確認する。
2. **SheetService の実装**: 既存スプレッドシートの「在庫移動リスト」「商品マスタ」等を読み書きする関数を実装し、Ref に相当する「JAN → 商品名」の解決を行う。
3. **メニュー＋1 画面**: メニューと「出荷」または「在庫移動」の 1 画面だけを実装し、doPost で 1 件登録できることを確認する。
4. **全画面の実装**: 入荷・棚卸・差異・商品情報を順に追加。
5. **トリガー・メール**: 在庫移動日報の Time-driven トリガーと MailJob を追加。
6. **現在庫・平均原価**: タスク 1・2 のロジックを InventoryLogic.gs に実装し、画面と紐付ける。
