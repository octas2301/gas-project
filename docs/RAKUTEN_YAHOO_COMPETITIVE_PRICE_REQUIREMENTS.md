# 楽天・Yahoo! 競合価格（セット別）取得 要件定義

## 1. 目的

- Amazon 情報を元に決めた**セット数構成**（マスタの「A.セット商品数」）に合わせて、**同じセット数**の楽天・Yahoo! の最安値を取得し、スプレッドシートにセットごと（行ごと）に書き出す。
- 楽天については、Yahoo! と同様の書き出し列（競合価格・URL・要確認・メモ）へ同じ種類の情報を出力する。
- セット数が API で取得できないため、**商品名**および**商品画像**から推測し、必要に応じて **GEMINI** で補完する。

---

## 2. スコープ

| 項目 | 内容 |
|------|------|
| 対象モール | 楽天市場（Ichiba Item Search API）、Yahoo! ショッピング（既存） |
| 対象シート | ▼商品マスタ(人間作業用) |
| 入力 | 選択行の JAN、A.セット商品数、必要に応じてメーカー名・商品名 |
| 出力 | 競合価格・競合URL・要確認・確認内容メモ（楽天は新規列を利用） |

---

## 3. 楽天：書き出し列（Yahoo! と同様）

マスタに以下の列を用意し、Yahoo! セット別取得と同様の情報を書き出す。

| 列名（想定） | 用途 |
|--------------|------|
| 競合価格楽天 | 当該行のセット数に一致する楽天の最安値（総量＋ポイント込み等、仕様で定義） |
| 競合URL楽天 | 上記最安商品の商品ページ URL |
| 要確認楽天 | 選択式（要確認／確認OK）。異常単価など要レビュー時に「要確認」 |
| 確認内容メモ楽天 | 要確認の理由（例：「5→2199円:1個あたりがAmazon競合価格比で異常に安い」） |
| 楽天価格設定 | 既存列。提案・手動設定用（本機能では更新しない想定） |

- 列名はスプレッドシートの実際のヘッダーと完全一致させる。定数はコード側で上記に合わせて定義する。

---

## 4. セット数の取得・推測

### 4.1 API の仕様

- **楽天** Ichiba Item Search API：セット数・数量の専用フィールドは**ない**。商品名・説明文のテキストから推測する。
- **Yahoo!** ショッピング API：同様にセット数フィールドは**ない**。商品名から推測する。

### 4.2 推測の優先順位と情報源

1. **商品名（テキスト）**：既存の `parseSetCountFromItemName`（正規表現）で抽出。取得できない場合に **GEMINI テキスト推測**を使用可能とする。
2. **商品画像**：画像 URL を Gemini Vision に渡してセット数を推測。楽天は `mediumImageUrls`、Yahoo! はヒットの画像 URL。
3. **画像推測時の解釈ルール（楽天・Yahoo! 向け）**：文字でセット数が書いてあればそれを優先（複数個が映っていなくても可）。文字がなければ同一パッケージの個数を数える。
4. **複数ソースの突き合わせ**：名前と画像の両方がある場合は一致時は採用、不一致時はルールで決定または要確認とする。

### 4.3 GEMINI 利用方針

- テキスト：正規表現で取れない場合に GEMINI で商品名からセット数推測。
- 画像：既存の画像推測（Vision）を楽天・Yahoo! にも適用し、プロンプトを「文字表記を優先」に調整。
- テスト段階では、楽天・Yahoo!・Amazon の候補を 1 シートに集約し、**モール横断の GEMINI 一括判定**で内容量と販売セット数を総合判断する。
- Amazon は既存実装の大幅改修は行わず、Keepa 等から取得した候補情報を**参照証拠**として利用する。
- 既存の単体テストメニュー（楽天単独・Yahoo!単独）は、統合テストの比較用として**当面残す**。

---

## 5. 楽天：セット別最安値の算出

- 選択行の JAN と A.セット商品数を取得。Ichiba API を呼び出し（JAN／メーカー＋商品名でフォールバック）。
- レスポンスの商品配列は `json.Items || json.Products || json.items` で取得。
- 各商品の商品名からセット数を推測し、行の A.セット商品数と一致したものだけを対象に「総量＋ポイント込み」を計算し最小値をその行の楽天最安値とする。
- 総量＋ポイント込み：総量＝販売価格＋送料（postageFlag で判定）、ポイント込み＝総量−ポイント還元（itemPrice×pointRate/100 等）。送料額が API で取れない場合は送料あり時は仮値または価格のみで比較。
- 要確認：1個あたりが競合価格 Amazon の一定倍率未満などは「要確認楽天」にし、確認内容メモ楽天に理由を記載。

---

## 6. 条件分岐の整理（楽天・Yahoo! 画像推測）

| 分岐 | 内容 |
|------|------|
| 画像内にセット数の文字あり | その数値を採用。 |
| 画像内に複数個の同一商品 | 個数をカウントして採用。 |
| 画像のみでは不明 | 商品名（正規表現／GEMINI）にフォールバック。 |
| 商品名と画像の両方あり・一致 | その値を採用。 |
| 商品名と画像の両方あり・不一致 | ルールで決定（画像の文字優先／単価で選択／要確認のいずれか）。 |
| 画像取得失敗・API エラー | 画像はスキップし、商品名のみで判定。 |

---

## 7. 切り戻し・調査のための方針

- **切り戻し**：既存の「選択行のJANで楽天から競合価格を取得」は変更しない。楽天セット別は**新規メニュー・新規関数**で追加し、不具合時はメニュー非表示や関数呼び出しを外すだけで切り戻し可能とする。
- **調査範囲の区切り**：セット数推測（商品名／画像／GEMINI）・API 呼び出し・価格計算・書き出しをそれぞれログで区別できるプレフィックス（例：`[楽天セット別]`）で出力する。
- **ログで改善**：JAN・検索キーワード・取得件数・セット数推定結果・採用価格・要確認理由などを Logger に残し、実行ログから原因切り分けや仕様改善ができるようにする。

---

## 8. 今後の拡張

- 画像ダウンロードは本要件の範囲外。列や保存先が決まれば別要件で追加可能。
- 楽天「商品価格ナビ製品検索API」は JAN で製品単位の価格取得に利用。セット別最安値は Ichiba Item Search API を主とする。

## 9. 切り戻し・ログ調査（実装メモ）

### 切り戻し手順

- **楽天セット別のみ止めたい場合**  
  - メニュー「選択行のJANで楽天からセット別競合価格を取得」の行をコメントアウトまたは削除する（`コード.js` 内の `addItem('選択行のJANで楽天からセット別競合価格を取得', 'menuFetchRakutenSetPricesToMaster')`）。  
  - 既存の「選択行のJANで楽天から競合価格を取得」と `menuFetchCompetitivePriceFromRakuten` は変更していないため、従来どおり利用可能。
- **定数だけ戻したい場合**  
  - `COL_COMPETITOR_URL_RAKUTEN` / `COL_REVIEW_STATUS_RAKUTEN` / `COL_REVIEW_MEMO_RAKUTEN` の 3 行を削除またはコメントアウト。  
  - `menuFetchRakutenSetPricesToMaster` 内でこれらの定数を参照しているため、メニューも同時に外すこと。

### ログ調査の目安（プレフィックス [楽天セット別]）

| ログの内容 | 調査の切り分け |
|------------|----------------|
| `API keyword=... 取得件数=N パース後=M` | API は成功しているが、itemPrice が無効な件が N−M 件ある |
| `API keyword=... HTTP 4xx/5xx` | 認証・レート制限・ネットワーク障害 |
| `フォールバック keyword=... 取得件数=0` | JAN では 0 件でメーカー+商品名でも 0 件（商品が Ichiba に無い or キーワード不一致） |
| `セット数不明 itemName先頭50字=...` | 商品名からセット数が取れていない（parseSetCountFromItemName のパターン追加 or GEMINI 推測の検討） |
| `送料あり セット数=...` | 送料込み価格が API で取れていないため、実質価格はポイント還元のみ。送料仮定値の要否を検討 |
| `JAN=... セット数別最安 セット数=A,B,C` | その JAN で検出したセット数と、そのセット数ごとの最安が計算されている |
| `行N セット数=X 該当なし` | マスタの A.セット商品数 X に一致する楽天商品が 1 件も無い（キーワード or セット数推測の見直し） |
| `行N セット数=X 価格=Y 要確認` | 1 個あたりが Amazon 競合価格の YAHOO_SET_PRICE_REVIEW_RATIO 未満で要確認にした |

### ③商品名一致スコア（プレフィックス [楽天商品名一致]）

- **定数（コード先頭付近）**: `RAKUTEN_NAME_MATCH_THRESHOLD`（採用閾値）、`RAKUTEN_NAME_MATCH_FULL_MATCH_SCORE` / `_WORD_MAX_SCORE` / `_MAKER_SCORE` / `_CHAR_MAX_SCORE`（各ルールの得点）、`RAKUTEN_NAME_MATCH_USE_FULL` / `_USE_WORD` / `_USE_MAKER` / `_USE_CHAR`（各ルールの有効フラグ）。変更するとロジックを調整できる。
- **ログで確認できること**: 閾値・有効ルール、除外されたヒットの score とどのルールが何点付いたか、採用されたヒットの score とルール内訳、JAN ごとの「取得 / 閾値以上採用 / 除外」件数。

| ログの内容 | 調査の切り分け |
|------------|----------------|
| `閾値=N 有効ルール=完全一致,単語一致,...` | 現在有効なルールと閾値。どのロジックが効いているか確認可能 |
| `除外 score=N ルール=単語一致:20 ... itemName=...` | スコア不足で除外されたヒット。ルール内訳で閾値・得点の見直しが可能 |
| `採用 セット数=X score=N ルール=...` | そのセット数の最安を更新したヒット。どのルールで採用されたか確認可能 |
| `JAN=... 取得=A 閾値以上採用=B 除外=C` | フィルタ前後の件数。B が 0 の場合は閾値引き下げ or 期待名・メーカー入力の確認 |

改善時は上記を頼りに「API 層」「セット数推測」「価格計算」「要確認判定」「商品名一致スコア」のどれに手を入れるか区切って変更できる。
