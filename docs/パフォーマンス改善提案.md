## 2. 修正が必要な箇所の提案

### 2.1 最優先：apiGetVarianceList（約7.4秒）

- **現象**: 差異タブを開くたびに 5〜7 秒かかっている。
- **想定原因**: 全商品の理論在庫・棚卸実数・差異を都度計算し、シート読み＋在庫計算が重い。
- **提案**:
  1. **キャッシュ**: 差異タブ表示結果を一定時間（例: 30秒）メモリに持つ。同一セッションでタブを切り替えただけなら再計算しない。
  2. **計算の遅延／段階表示**: まず「読み込み中」を出し、バックグラウンドで差異リストを取得。取得できた件数分から順に表示する（ページングや仮想スクロールの検討）。
  3. **サーバー側の最適化**: `getVarianceList()` でシート読み回数を減らす（一括 getValues で必要な列だけ取得、ループ内の getCurrentStockByJan 呼び出しをまとめて計算するなど）。

### 2.2 高：apiReceiving（約2秒）

- **現象**: 入荷登録1回あたり約2秒。
- **想定原因**: 入荷行の追加に加え、平均単価更新・ログ記録など複数シート／複数書き込みが発生している。
- **提案**:
  1. 入荷・出荷・移動など「1行追加」系は、可能な限り **1回の getRange + setValues** でまとめて書く（バッチ書き込み）。
  2. 平均単価更新を別APIに分けず、同じ処理内でまとめて読み書きする（読み1回・書き1回に寄せる）。

### 2.3 高：apiCountAdjust（約1.8秒）

- **現象**: 棚卸確定後修正の保存に約1.8秒。
- **想定原因**: 確定後修正行の追加とログ記録、およびシートの取得・追記のオーバーヘッド。
- **提案**:
  1. `appendCountAdjustRow` 内でシート取得を1回にし、必要なら同じシートに対して複数行をまとめて append する設計にする。
  2. ログ記録を「パフォーマンスログ」と「操作ログ」で別シートにしている場合、書き込み回数を減らす（例: 操作ログをバッファしてから一括追記）は検討余地あり。

### 2.4 中：apiLookupProduct（約0.3〜1.2秒、変動大）

- **現象**: JAN入力（blur）のたびに 300〜1200ms 程度。回数が多いと体感が重い。
- **想定原因**: 商品マスタ検索＋在庫計算の都度実行。
- **提案**:
  1. **クライアント側キャッシュ**: 同じ JAN を連続で入力した場合は前回結果を再利用（デバウンス＋キャッシュ）。
  2. **サーバー側**: 商品マスタを「単品JAN」でインデックス化して検索を1回の getValues で済ませる（既にそうなっていれば、読み範囲を列単位で絞る）。

### 2.5 中：apiGetCountSession（約0.9〜1秒）

- **現象**: 棚卸タブ表示・更新のたびに約1秒。
- **想定原因**: 棚卸セッションシートの読み取りと最終行の検索。
- **提案**:
  1. セッション情報を PropertiesService などに短時間キャッシュする（同一リクエスト内で複数回呼ばれないようにする、またはクライアント側で1回だけ取得して使い回す）。

### 2.6 その他（apiGetCurrentStockList など）

- **apiGetCurrentStockList** は今回ログにないが、過去計測で 2.5〜4.3 秒程度出ている想定。在庫タブを開くたびに全商品の現在庫を計算しているため、差異と同様「キャッシュ」「段階表示」「サーバー側の一括計算」が有効。

## 3. 実装の進め方（推奨順）

1. **apiGetVarianceList** のキャッシュまたは段階表示（クライアント＋サーバー）。
2. **apiReceiving** / **apiCountAdjust** の書き込み回数削減（バッチ化）。
3. **apiLookupProduct** のクライアント側デバウンス＋キャッシュ。
4. **apiGetCountSession** のキャッシュまたは呼び出し削減。
5. **apiGetCurrentStockList** のキャッシュまたは段階表示（必要に応じて）。

## 5. クリック・タップ対象の範囲（UI）

### 5.1 修正内容（index.html）

- **ボタン（.btn）**  
  - **対象範囲**: ボタン全体（幅100％・最小高さ 48px・padding 14px の四角全体）がクリック／タップ対象です。  
  - **変更点**: `min-height: 48px` と `display: inline-block` を追加し、縦方向の当たり判定を広くしました。文字の周りだけでなく、ボタンの縁まで押して反応するようにしています。
- **ナビ（.nav-item）**  
  - **対象範囲**: 各タブ（出荷・移動・入荷…）の **1つ1つのタブ全体** がクリック／タップ対象です。  
  - **変更点**: `min-height: 44px`、`padding: 12px 8px`（従来は 8px 4px）に変更し、タップしやすいサイズにしました。`display: flex` でアイコンと文字を中央寄せしたまま、タブの端から端まで反応するようにしています。

### 5.2 技術的なクリック／タップ範囲

- **ボタン**: `<button>` または `onclick` の付いた要素の **表示されている矩形全体**（padding を含む）がクリック可能領域です。マウスならその矩形内、タッチならその矩形内のタップで反応します。
- **ナビ**: `<a class="nav-item">` の **表示されている矩形全体**（padding を含む）が同様にクリック／タップ可能です。  
「縁や左右にずらすとクリックできる」と感じていた場合は、以前は padding が 8px 4px と小さく、見た目の文字部分だけが狭く反応していた可能性があります。上記の変更で、**タブの四角全体**が反応するようになっています。
