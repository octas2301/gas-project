# フロー接続・必須要件・クリティカルパス・自動化優先順位

細かい要件定義は**各フローを構築する際に詰める**。ここでは前後のフローの繋がりから**必須な要件**と**クリティカルパス**だけを今回の整理で決め、**自動化構築の優先順位**をすり合わせの土台とする。

---

## 1. フロー間の接続と必須要件

後工程が始められるために、各接続で「ここだけは揃っていること」を必須とする。

| 接続（前 → 後） | 必須な要件（今回決める最小限） | 細かくするタイミング |
|------------------|--------------------------------|----------------------|
| **リサーチ・見積もり → 出品情報** | 商品を一意に識別するキー（JAN / ASIN / 商品名のいずれか）、仕入単価・仕入先が分かること。出品するかどうかの判断結果が渡ること。 | 各フロー構築時 |
| **出品情報 → 各モール出品** | モール共通で必須な項目：商品識別子、商品名、価格、在庫数、画像（1枚以上）、説明・キャッチ文。モール別の追加必須項目は各モールのフロー構築時に定義。 | 楽天・Yahoo! は HANDOVER 参照。Amazon は構築時 |
| **出品 → 販売実績** | モール・商品（またはSKU）・日付をキーに、売上・数量・手数料が取得・集計できること。 | 実績取得フロー構築時 |
| **販売実績・在庫 → 商品管理** | 実績と在庫数が時系列で紐づけられ、発注判断のトリガーに使えること。 | 発注・在庫フロー構築時 |
| **出品・販売 → 顧客対応** | 問い合わせ・レビューがモール・商品（または注文）と紐づけて把握できること。 | 顧客対応フロー構築時 |
| **販売・仕入・経費 → 財務・経理** | 売上・振込・手数料・仕入・経費がモール別・月別に集計できること。 | 経理フロー構築時 |

## 2. クリティカルパス

**商品が売れるまで**の経路がボトルネックになる。

```
リサーチ・見積もり → 商品出品情報の作成 → 出品（各モール）
```

- この順で「何を渡すか」が決まっていないと後工程が始められない。
- 特に **「商品出品情報」の最小必須項目** と **誰がいつ入力するか（入力元）** だけは今回の整理で決める。
- 上記以外（項目の詳細、バリデーション、UI など）は各フロー構築時に詰める。

**今回の整理で決めたこと**

- リサーチ・見積もり → 出品情報：商品識別子（JAN/ASIN/商品名）、仕入単価・仕入先、出品可否が渡ること。
- 出品情報 → 各モール：商品識別子、商品名、価格、在庫数、画像（1枚以上）、説明・キャッチ文が共通必須。
- 細かい要件定義は各フロー構築時に詰める旨を本ドキュメントで明文化する。

## 3. 自動化構築の優先順位（すり合わせ用）

クリティカルパスと依存関係を踏まえた案。番号が小さいほど先に着手する想定。すり合わせで入れ替え・分割可。

| 優先順位 | 対象 | 内容の要約 | 理由 |
|----------|------|------------|------|
| 1 | 基盤 | 要件・runbook・本ドキュメント（フロー接続・必須要件・クリティカルパス）の整備 | 以降の判断の土台 |
| 2 | 商品出品情報の整備 | 最小必須項目の定義と、その入力元を1つ確保する（手入力/CSV/既存マスタ連携など）。既存 gas-project（楽天・Yahoo!）はここから参照 | クリティカルパス上のボトルネック。ここが無いと出品が増やせない |
| 3 | 出品の自動化 | 楽天・Yahoo! は既存運用を継続。Amazon は範囲を決めてから構築 | 出品情報が整ってから意味がある |
| 4 | リサーチ・見積もりの効率化 | 見積比較・商品リサーチの下書きなど、出品情報への投入を楽にする部分を優先 | クリティカルパスの上流を効率化 |
| 5 | 販売実績の取得 | モール別の実績取得の自動化（集計・可視化の範囲は構築時に決定） | 出品が回り始めたら必要 |
| 6 | 商品管理 | 発注アラート・在庫の可視化。需要予測などは実績データが溜まってから検討 | 実績・在庫データに依存 |
| 7 | 顧客対応 | 問い合わせ・レビュー返信の文案支援・振り分け | 出品・販売が回った後に効果が大きい |
| 8 | 財務・経理 | 振込・手数料の取り込みと月次集計 | 販売・仕入データに依存 |

## 4. 優先順位2の具体化（商品出品情報の整備）

構築時に詰める前提で、入力元の選択肢と最小必須項目を1枚にまとめる。

**入力元の選択肢（先に使うものを1つ確保する）**

| 入力元 | 内容 | 備考 |
|--------|------|------|
| 手入力 | スプレッドシート等に人が直接入力 | 初期はこれで開始可能 |
| CSV取込 | セラセン・問屋等のCSVを取込んで項目マッピング | 商品出品情報フォルダ内CSVを参照する仕様は HANDOVER §10 を参照 |
| 既存マスタ連携 | ▼商品マスタ(人間作業用) や AI情報取得data から連携 | 既存 gas-project のマスタ・AI情報取得data を参照 |

**採用**: **既存マスタ連携**（▼商品マスタ・AI情報取得data から商品出品情報へ連携する形で整備する）。

**最小必須項目一覧（出品情報 → 各モールに渡す共通必須）**

| 項目 | 内容 |
|------|------|
| 商品識別子 | JAN / ASIN / 商品名のいずれかで一意に識別できること |
| 商品名 | 各モールの文字数制限内で設定 |
| 価格 | 販売価格（税込/税抜はモール別に構築時定義） |
| 在庫数 | 出荷可能な数量 |
| 画像 | 1枚以上。形式・枚数上限はモール別に構築時定義 |
| 説明・キャッチ文 | 商品説明およびキャッチコピー（モール別の項目は構築時定義） |

上記以外の項目（送料・手数料・カテゴリ・キーワード等）は各フロー構築時に追加する。

**既存マスタ連携による最小必須項目の対応（入力手段を確保した結果）**

既存の ▼商品マスタ(人間作業用) と AI情報取得data を入力元とする場合、最小必須項目は次の列で確保できる。楽天・Yahoo 出力は既にこれらの列を参照している（コード.js の getMVal・マスタ列マップ、HANDOVER §11 参照）。

| 最小必須項目 | マスタの列（既存） | AI情報取得data（既存） | 備考 |
|--------------|--------------------|-------------------------|------|
| 商品識別子 | 親SKU、子SKU、ASINコード | ASINコード、JANコード、商品名 | 突合は ASIN/JAN/商品名。syncAiDataToMaster が親行のみ更新 |
| 商品名 | 商品名 | 商品名 | 子SKU行は「商品名【バリエーション値】」で展開される想定（既存ロジック） |
| 価格 | 商品の販売価格（列がある場合）または ▼マスタ 経由 | — | マスタに列が無い場合は構築時に追加。NUMERIC_COL_NAMES に含まれる |
| 在庫数 | 在庫数（列がある場合）または ▼マスタ 経由 | — | 同上 |
| 画像 | ▼マスタ(参考情報(画像URL)) | 参考情報(画像URL) | AI→マスタ同期で親行に反映。楽天・Yahoo はマスタから参照 |
| 説明・キャッチ文 | ▼マスタ(商品説明)、▼マスタ(楽天キャッチ(60-80字))、▼マスタ(Yahooキャッチ(20-30字)) 等（COMPARE_ITEMS のうち説明・キャッチ系） | 同項目名 | AI情報取得data で生成 → syncAiDataToMaster でマスタ親行へ反映。出品時はマスタから参照 |

**実施内容**: 入力元＝既存マスタ連携を採用したうえで、上記のとおりマスタ・AI情報取得data の既存列で最小必須項目を賄えることを整理した。実装時は既存の syncAiDataToMaster・楽天CSV出力・Yahoo の YahooDataBuilder が参照している列を変更せず、不足列のみマスタまたはマッピングで追加する。

**参考資料（現状の項目確認用）**: プロジェクト内フォルダ **`参考資料：出品自動化スプシデータ`** に、現時点のマスタシート等の情報をCSVで保存したファイルがある。このCSVの項目で Amazon・楽天・Yahoo! に必要な情報を揃えている。確認時は次のファイルを参照する。
- `自動出品【通常260221】●月●日（●様）必要sheet）.csv` … AI情報取得data 相当の列（JAN, ASIN, 商品名, 参考情報(画像URL), 各COMPARE_ITEMS の [Gemini][GPT][正解]、▼マスタ(...)）を含む。
- `自動出品【通常260221】●月●日（●様）必要sheetCSV形式違い）.csv` … マスタ・出品用の列（商品管理番号, 販売価格, 在庫数, 商品コード, 商品名, 対象ASIN, 画像URL 等）を含む。モール別の出力項目も含まれる。

## 8. 優先順位3以降の進め方（続き）

優先順位2まで完了したあとの「続き」は次のとおり。

**次に確認すること（優先順位3）**

- **楽天・Yahoo!**: 既存の gas-project（一括出品・CSV・画像API・完了メール等）を**そのまま継続運用**する。変更不要。
- **Amazon**: **要件だけ整理する（選択肢C）**。実装は後日。以下に必須項目・フローを文書で整理する。

**Amazon の要件整理（実装は後日・選択肢C）**

- **詳細**: [AMAZON_REQUIREMENTS.md](AMAZON_REQUIREMENTS.md) にフロー・共通必須項目・Amazon 固有項目・広告の扱いを整理した。
- **フロー**: マスタ／AI情報取得data（既存マスタ連携）→ 商品出品情報（最小必須項目を揃える）→ **Amazon 出品**（リスティング登録・必要に応じ広告）。楽天・Yahoo と同様、商品出品情報を一元化したうえでモール別に出力する。
- **共通必須項目**: §4のとおり。価格はマスタの **販売価格amazon** 列を使用（Phase 2 確定）。既存の参考資料CSV（`参考資料：出品自動化スプシデータ`）の「必要sheetCSV形式違い」に、対象ASIN・商品名・販売価格・在庫数・画像URL 等の列があり、Amazon 用の項目も含まれている。
- **Amazon 固有で必要な項目（実装時に詰める）**: 商品管理番号（SKU）、商品コード（JAN/EAN等）・商品コードのタイプ、コンディション、リードタイム、カテゴリ（Product カテゴリー）、フルフィルメント（FBA/自己発送）、画像URL（メイン・サブ）、その他プラットフォーム必須項目。参考資料CSVの列名を実装時のマッピング基準とする。
- **広告**: リスティング登録後に、スポンサード商品等の範囲は別途要件化する。今回は「出品（リスティング）の必須項目・フロー」までを文書で整理し、実装は後日とする。

**既存マスタ連携の整備で実装時にやること（HANDOVER §10 ベース）**

- マスタ・AI情報取得data の既存列で最小必須項目を賄う（§4の対応表のとおり）。不足列があればマスタまたはマッピングで追加。
- 必要に応じ **調査項目CSV** を `商品出品情報/` または `参考資料：出品自動化スプシデータ` に置き、取得項目・マスタ列対応を定義する。
- コード側: 既存の **syncAiDataToMaster**・楽天CSV出力・Yahoo の参照を壊さない範囲で、Config / MasterSheet / ProductInfoAcquisition 等のファイル分割を検討する（§10.4）。まずは既存 コード.js・Yahoo.js のまま運用し、リファクタは必要になったタイミングで行う想定でよい。

**実装タスクの具体化**: 上記を実行するためのタスク一覧は [MASTER_LINKAGE_TASKS.md](MASTER_LINKAGE_TASKS.md) にまとめた。Phase 1（現状確認・不足の洗い出し）→ Phase 2（データ・マッピングの整備）→ Phase 3（コード側の対応は必要時のみ）の順で進める。

**進める順序の提案（マスタ連携 vs リサーチ）**

- **先に進めるもの**: **既存マスタ連携の実装タスクの具体化**。
- **理由**: リサーチで得た「競合価格・セット数」を元にした出品価格・セット数のAI提案は、**書き込み先がマスタ／商品出品情報**になる。その形（列・フロー）が決まっていないと、リサーチの出力項目や連携方法を決められない。マスタ連携の具体化で「価格・セット数をどこにどう渡すか」を決めてから、リサーチ項目・取得方法を整理した方が無駄が少ない。
- **リサーチの実装順**: マスタ連携の具体化の**後**。まずは下記のとおり「項目の違い」「取得方法の選択肢」を文書で整理するだけにとどめ、実装はマスタ連携が固まってから進める。

**リサーチの整理方針（文書で整理する範囲）**

- **詳細**: [RESEARCH_AND_ESTIMATE.md](RESEARCH_AND_ESTIMATE.md) に、最終イメージ・出品時 vs 仕入れ時のリサーチ項目・取得方法の選択肢（API/Keepa/スクレイピング等）・見積もりの整理を記載した。
- **今後の方針（確定）**: 同 doc「§0 今後の方針」を参照。AIで埋められるものはすべてAIで埋め、人間は提案が気に入らないときのみ修正。初回登録はほぼAI提案。既存「2. 全データ一括生成」は残さず、新フローに置き換える。使わなくなる既存仕組みが出る想定。
- **最終イメージ**: リサーチで競合価格・セット数を調べ → その価格を元に**出品価格・セット数をAIで提案** → マスタ／商品出品情報に反映して出品する流れを目指す。
- **出品時に必要なリサーチ項目** と **仕入れ時に必要なリサーチ項目** は厳密には異なる。両方の項目を分けて定義し、共通部分と用途別の差分を文書で整理する（実装時に参照する）。
- **リサーチ方法**（API連携 / 外部Keepa / スクレイピング 等）で**進める工程が変わる**。選択肢ごとに「取得できる項目・コスト・実装工数」を比較できるよう、選択肢を文書で整理する。どの方法で進めるかは、マスタ連携の具体化や運用方針が固まってから決定する。
- 上記（項目の違い・取得方法の選択肢）の整理がまだであれば、**他の確定できる項目を先に進める**。リサーチの詳細な工程設計は、マスタ連携の具体化のあとに回す。

**優先順位4以降**: リサーチ・見積もりの効率化（上記のとおり項目・取得方法を整理したうえで実装）→ 販売実績取得 → 商品管理 → 顧客対応 → 財務・経理の順で、必要に応じて要件を詰めながら進める。

**開発効率優先の実装順（ユーザー確認 4-6 反映・任せます）**

効率よく開発が進められる順に並べた。実装はこの順で進める。

| 順 | 内容 | 備考 |
|----|------|------|
| 1 | **基盤の確認** | 要件・Runbook・本ドキュメントの該当箇所を読んで承認（§6の表を参照）。 |
| 2 | **ASIN貼り付け（Keepa用）シート＋GAS連携** | 人間がASINを貼る→GASがKeepaで取得。シート名・列名・メニューを整備。 |

**確認回答の反映（本チャット・2026-02）**

- **N-2（一連）で進める**。Keepa 取得 → セット数・価格提案 → マスタ反映 → 親行・子行自動作成まで一連実行。**「親行・子行だけ再作成」オプションを用意**する。
- **対象行**: ▼商品マスタ(人間作業用) で **◎が付いている行**。
- **実行順**: (1) Keepa取り込み（GAS）→ (2) 人間が評価を修正（セット数が間違っていないか・対象品と合致しているかのみ確認）→ (3) その後のフローを実行（GAS: 競合セット数・穴のセット数提案 → 価格・セット数提案をマスタに反映。必要なら親行・子行自動作成）。
- **セット数提案の出力先**: **マスタの固定列に直接書く**（ASIN貼り付けシート右端2列ではない）。
- **セット数ごとの最低価格**: **GASで算出**する。人間は価格の優劣判断をしない。
- **次の開発優先**: 優先4（セット数候補・価格設定）と優先5（マスタ親行・子行自動作成）は **効率のよい順**で実装してよい。
