# Keepa分析・モール横断前倒し 要件定義

**目的**: 後から開発を再開できるよう要件を固定し、引き継ぎ可能な形で残す。

---

## 1. 概要とスコープ

### 1.1 目的

- Amazon（Keepa）のセット数判定精度を、楽天・Yahooの候補を**前倒しで取得**し「3モールの文脈」で**内容量 vs セット数**を判定することで向上させる。
- あわせてモール横断実行時の**楽天・Yahoo API 二重呼び**を防ぐ。

### 1.2 背景

- 現状は Keepa 取得時は商品名・画像のみでセット数推定（正規表現／AI）。楽天・Yahoo はモール横断で取得し統合判定しているため、Amazon だけ精度が落ちやすい。
- P表記に限らず「多くの情報」から内容量とセット数を区別したい。
- 運用方針として、マスタのセット数上書き（案C）は不採用。本要件は**Keepaのセット数推定精度向上**と**モール横断のAPI重複防止**に限定する。

### 1.3 対象

| 対象 | 処理・シート |
|------|----------------|
| Keepa取得 | ASIN貼り付け（Keepa用）での Keepa 取得フロー。`runKeepaFetchAsinPasteSheetImpl`（コード.js 5611行付近）。 |
| モール横断 | モール横断 セット数統合判定。`menuTestCrossMallSetCountJudge`（コード.js 2832行付近）。 |

### 1.4 対象外

- マスタの A.セット商品数（▼マスタ(総個数)）や競合価格amazonの上書き（案Cは不採用）。
- 既存の「モール横断結果を楽天・Yahoo! マスタに反映」で楽天・Yahooのみ書く仕様は変更しない。

## 2. 機能要件

### 2.1 Keepa分析時の前倒し（ブロック単位）

- ブロックの **expectedJAN**（コード.js 5722行付近 `productNameRow[nameCol + 1]`）が有効（8桁以上）なとき、**そのJANで楽天・Yahooの候補を取得**する（モール横断の一部を前倒し）。
- 取得した楽天・Yahoo候補と、**当該ブロックのKeepa結果（Amazon候補）**を合わせて、**モール横断と同様の統合判定**（`inferCrossMallSetCountsByGemini` 相当）を実行する。
- 統合結果を用いて、**各Keepa行（ASIN）のセット数**および評価（◎等）を決定する。既存の `resolveSetCountByUnitPrice` や評価ロジックに渡す入力の質を上げる。
- 同一実行内で同じJANが複数ブロックに現れた場合は、**そのJANの楽天・Yahoo取得は1回だけ**とし、2ブロック目以降はキャッシュ参照とする。

### 2.2 モール横断実行時のキャッシュ利用

- `menuTestCrossMallSetCountJudge` で各JANを処理する際、**当該JANの楽天・Yahoo候補がキャッシュに存在する場合はAPIを呼ばずキャッシュを利用**する。
- キャッシュに無いJANは現行どおり楽天・Yahoo API を呼ぶ。
- Amazon候補は従来どおりマスタのそのJANのASINでKeepaを呼ぶ（**重複排除は楽天・Yahooに限定**する方針でよい）。

### 2.3 双方向キャッシュ

- **Keepa分析が先**の場合: ブロックの expectedJAN で楽天・Yahooを取得 → 統合判定 → 結果でKeepaのセット数・評価を更新 → そのJANの楽天・Yahoo候補（および必要なら統合結果）をキャッシュに保存。
- **モール横断が先**の場合: そのJANの楽天・Yahooを取得したらキャッシュに保存する。後からKeepa分析で同じJANのブロックを処理するときは、キャッシュにデータがあれば楽天・Yahooを再取得せずキャッシュで統合判定する。

---

## 3. キャッシュ仕様

| 項目 | 内容 |
|------|------|
| **キー** | JAN（文字列、トリム済み）。 |
| **値** | 楽天・Yahooの候補リスト（`menuTestCrossMallSetCountJudge` 内で組み立てている `candidates` のうち source が `rakuten` / `yahoo` のもの）。必要に応じて統合セット数付きの候補、またはモール横断シート24列形式の行配列を保存する（**実装時にどちらで持つか決定**）。 |
| **保存先** | 専用シート（例: 「モール横断キャッシュ」「JAN別候補キャッシュ」）を推奨。Script Properties / CacheService は容量制限に注意。シートの場合は「JAN」「取得日時」「データ（JSONまたは行配列）」等の列構成を定義する。 |
| **TTL（有効期限）** | 例として24時間。キャッシュがTTLを超えている場合は「無い」とみなし、該当JANはAPI再取得する。TTLの値は定数または Script Property で設定可能にするとよい。 |

---

## 4. 実行順序とAPI重複排除

- 重複排除の対象は**楽天・Yahoo API に限定**する。
- Keepaは「マスタのASIN」と「ブロックのASIN」が一致するとは限らないため、両フローで必要に応じて呼ぶことを許容する。
- キャッシュの**書き込み**は (1) Keepa分析で楽天・Yahooを取得したとき、(2) モール横断で楽天・Yahooを取得したとき、の**両方**で行う。
- キャッシュの**読み込み**は (1) Keepa分析でそのブロックの expectedJAN のデータが必要なとき、(2) モール横断でそのJANの楽天・Yahooが必要なとき、の**両方**で行う。

---

## 5. comparisonCtx（比較前提）の扱い

- モール横断では `getCrossMallComparisonContext(ss, masterValues, masterColMap, headerRowIdx, dataIdx, jan)` で送料・手数料率・原価等を取得している（コード.js 内）。
- Keepa分析時はブロックに対応するマスタ行が一意でない場合があるため、**ブロックの卸値**（`getCostTaxInForAsinPasteBlock`）や可能な範囲でマスタ・設定を参照し、統合判定の品質がモール横断に近づくよう comparisonCtx を組み立てる方針とする。
- 実装時にマスタ行との対応が取れるかどうかも検討する。

---

## 6. 非機能要件・制約

- **既存フロー破壊禁止**: Keepa取得・モール横断の既存の「キャッシュなし」パスはそのまま動くこと。キャッシュは追加オプションとする。
- **JANが無いブロック**: expectedJAN が無効または空のブロックは、現行どおりKeepaのみのセット数推定（`extractSetCountFromTitle` / `inferSetCountsByGemini` 等）にフォールバックする。
- **エラー時**: 楽天・Yahoo前倒し取得や統合判定が失敗した場合、そのブロックは従来どおりKeepaのみの結果で続行する（全体を止めない）。

## 8. テスト観点

- **実行順（Keepa先）**: Keepa分析のみ実行 → 該当JANのモール横断実行時に楽天・Yahooが再取得されていないこと（ログまたはネットワークで確認）。
- **実行順（モール横断先）**: モール横断を先に実行 → 同じJANのブロックでKeepa分析時に楽天・Yahooが再取得されていないこと。
- **同一JAN複数ブロック**: 1回のKeepa取得実行で同じ expectedJAN のブロックが複数あるとき、楽天・Yahooは1回だけ取得されること。
- **キャッシュTTL**: 有効期限経過後は該当JANでAPIが再呼びされること。
- **フォールバック**: expectedJAN が無いブロック、またはキャッシュ・統合判定エラー時は、現行どおりKeepaのみの結果で処理が完了すること。
- **評価・セット数**: 前倒し＋統合判定を有効にした場合、該当ブロックのKeepa行のセット数・評価（◎）が、現行より妥当になることを目視またはサンプルで確認する。

## 実装時の手順（引き継ぎ用）

1. 本要件定義書（`docs/Keepa分析_モール横断前倒し_要件定義.md`）を開く。
2. コード.js の以下を確認する。
   - `runKeepaFetchAsinPasteSheetImpl`（5708行付近）
   - `menuTestCrossMallSetCountJudge`（2832行付近）
   - `inferCrossMallSetCountsByGemini`（2595行付近）
   - 候補組み立てのループ（2940–3102行付近）
3. §3 のキャッシュ保存先（シート名・列）を決定し、キャッシュ読書の共通関数を追加する。
4. §2.1 に従い Keepa ブロック内で expectedJAN → キャッシュ参照 or 楽天・Yahoo取得 → 統合判定 → セット数・評価への反映 → キャッシュ書き込みを実装する。
5. §2.2 に従い モール横断で JAN ごとにキャッシュ参照を行い、あれば楽天・Yahoo取得をスキップしてキャッシュから候補を復元し、キャッシュ書き込みを追加する。
6. §8 のテスト観点で動作確認する。
