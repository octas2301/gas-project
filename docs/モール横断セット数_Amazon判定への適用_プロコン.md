# モール横断セット数をAmazonの判定にも使う場合のプロコン・提案

## 現状フロー（比較基準）

- **Amazon（Keepa）**: ASIN貼り付けシートでKeepa取得 → 商品名・画像からセット数推定（正規表現／画像AI／参考画像）→ 競合価格Amazonは送料込み最安（NEW_FBM_SHIPPING）を取得してマスタに反映。
- **楽天・Yahoo!**: モール横断 セット数統合判定 → モール横断結果をマスタに反映。セット数はGemini統合判定で高精度。
- **価格・セット数提案**: マスタの競合価格（Amazon/楽天/Yahoo!）と卸値から、選択行に価格・セット数提案を反映（`proposePriceAndSetFromCompetitive`）。**セット数はマスタの「▼マスタ(総個数)」を参照**し、既に決まっている前提で単価比較・価格提案を行う。

つまり、**現状は「Amazonのセット数はKeepa取得フローで決まり、楽天・Yahoo!のセット数はモール横断で決まる」**という二系統になっている。

---

## 案A: 現状維持（AmazonはKeepaのみでセット数判定）

| 観点 | 内容 |
|------|------|
| **メリット** | 実行順に依存しない。Keepa取得だけすれば価格・セット数が揃う。既存運用のまま。 |
| **デメリット** | 同一JANで楽天・Yahoo!はモール横断の高精度セット数、Amazonだけ商品名／画像由来で精度が落ちうる。3モールでセット数が食い違う可能性。 |
| **不具合リスク** | 低。変更なし。 |

---

## 案B: モール横断のセット数をAmazonの判定にも適用する

**やり方**: モール横断実行後に、同一JANのマスタ行（レ点行）について、**▼マスタ(総個数)** をモール横断シートの「統合セット数」で上書きする、または別列「モール横断セット数」を書き、価格提案時にそれを優先して使う。

| 観点 | 内容 |
|------|------|
| **メリット** | 3モールでセット数を統一でき、単価ベースの比較・提案が一貫する。モール横断のGemini判定は文脈が豊富で精度が高い。 |
| **デメリット** | **実行順が固定される**（先にモール横断を実行しないとAmazon側のセット数が更新されない）。モール横断を実行しない運用だと、Amazonは従来どおりKeepa由来のままになり、**データの出所が二系統混在**する。 |
| **不具合リスク** | 中。同一JANの親・子・セット数違いの行と、モール横断シートの「どの行をどのマスタ行に当てるか」の対応を間違えると、**別セット数にセット数がずれる**。反映先列（▼マスタ(総個数)を上書きするか、別列のみか）をはっきり決めないと、上書きの二重実行や意図しない上書きがあり得る。 |

---

## 案C: ハイブリッド（モール横断を実行したJANだけAmazonにも適用）

**やり方**: モール横断結果をマスタに反映する処理のなかで、**楽天・Yahoo!の価格・URLに加え、同一JANのマスタ行の「▼マスタ(総個数)」を、モール横断シートのそのJANの代表統合セット数（例: 最安候補の統合セット数）で上書きする**。モール横断を実行していないJANは従来どおりKeepaのみ。

| 観点 | 内容 |
|------|------|
| **メリット** | モール横断をやったJANだけ3モール統一になり、実行順も「モール横断 → 反映」に揃えればよい。運用が分かりやすい。 |
| **デメリット** | 「モール横断をしたJAN」と「していないJAN」でセット数の出所が変わり、運用説明が必要。代表セット数の決め方（1行目採用／最安の統合セット数など）を仕様で固定する必要がある。 |
| **不具合リスク** | 中。代表セット数の取り方や、親子・セット数別の行への振り分けを誤ると、一部行だけセット数がずれる。 |

---

## 厳しい評価と提案

- **案A（現状維持）**: リスクは最小だが、3モールでセット数が揃わず、単価比較の解釈がモールごとに変わる。
- **案B**: 精度と一貫性は最も良いが、**実行順の強制**と**モール横断未実施時の二系統混在**が運用負荷になる。厳しく見ると「必ずモール横断から入る」運用にしないと効果が半減する。
- **案C**: モール横断をしたJANに限定してAmazonにも適用するため、**運用の線引きが明確**で、案Bより現実的。

**提案**: **案C（ハイブリッド）を推す。**

- 運用は「レ点を付ける → モール横断 セット数統合判定 → モール横断結果をマスタに反映」に統一する。
- 「モール横断結果をマスタに反映」の処理で、**楽天・Yahoo!に加え、同一JANのレ点行の「▼マスタ(総個数)」を、そのJANのモール横断結果から求めた代表統合セット数で上書きする**（代表の決め方: 例として「統合セット数が有効な候補の最頻値」または「価格+送料が最安の候補の統合セット数」を仕様で固定）。
- これにより、モール横断を実行したJANについてはAmazonも含めセット数が揃い、価格・セット数提案の単価比較が意味を持つ。モール横断をしないJANは従来どおりKeepaのみで、混在は「モール横断の有無」で説明できる。

**採用しない場合**: 現状のまま**案A**とし、Amazonのセット数はKeepa由来のみにしておく。その場合は「3モールでセット数が完全には揃わない」ことを仕様として明示する。

## 他案（参考）

- **別列のみ書き、上書きしない**: モール横断で「モール横断セット数」列にだけ書き、価格提案時は「▼マスタ(総個数)が空ならモール横断セット数を使う」などフォールバックにすると、既存の▼マスタ(総個数)を壊さずに試せる。運用が複雑になる代わりに、ロールバックが容易。
