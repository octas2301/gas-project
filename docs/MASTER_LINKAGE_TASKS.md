# 既存マスタ連携の実装タスク一覧

既存の ▼商品マスタ(人間作業用) と AI情報取得data を入力元として、商品出品情報（最小必須項目）を揃えるための実装タスクを具体化した一覧。  
前提は [FLOW_AND_PRIORITY.md](FLOW_AND_PRIORITY.md) §4「既存マスタ連携による最小必須項目の対応」および [HANDOVER.md](../HANDOVER.md) §10。

---

## タスクの進め方

- **楽天の既存ロジック（generateRakutenCSV 等）は変更・干渉しない**（HANDOVER §0）。
- 既存の syncAiDataToMaster・楽天CSV出力・Yahoo の YahooDataBuilder が参照している列を変更せず、**不足列のみ**マスタまたはマッピングで追加する。
- 参考資料: `参考資料：出品自動化スプシデータ` のCSVで現状の列を確認する。

## Phase 1 実施結果（2026-02 実施）

### 1.1 マスタ・AI情報取得data の列の現状確認

**AI情報取得data（参考資料「必要sheet）.csv」）**

- **ある**: 商品識別子（JANコード, ASINコード, 商品名）、画像（参考情報(画像URL)、▼マスタ(参考情報(画像URL))）、説明・キャッチ（各COMPARE_ITEMS の [正解] および ▼マスタ(...) 列）。卸値(税抜), 卸値(税込)。
- **ない**: 「商品の販売価格」という列名はない（卸値はある）。「在庫数」はない。AI情報取得data は親行向けのため、在庫はマスタ側で持つ想定でよい。

**マスタ（参考資料「必要sheetCSV形式違い）.csv」・行4がヘッダー相当）**

- **ある**: 商品管理番号, 商品コード(JANコード等), 商品名, 対象ASIN, 出品者SKUのメイン画像URL 等。楽天・Yahoo・Amazon 向けの項目が含まれる。
- **価格**: 商品の販売価格を**各モールごとに**設定している。必要に応じて列名にモール名を付与可（例: 楽天の販売価格, Yahooの販売価格）。
- **在庫数**: 「在庫数」で存在する。各モールごとに設定しているが、出品時は共通でよい。

**不足列リスト（要確認後の結論）**

| 項目 | 状況 | 対応 |
|------|------|------|
| 価格 | モール別列あり。コード参照列の対応は 1.3 で方針化 | Phase 2 で参照方法（共通1列かモール別列名指定か）を決定する |
| 在庫数 | 存在。出品時は共通でOK、追加不要 | 列名「在庫数」1本で足りる |

### 1.2 楽天・Yahoo 出力が参照している列の確認

**コード.js（楽天CSV出力・マスタ読込）**

- **固定列**: 親SKU, 子SKU, 出品CK, ASINコード, JANコード, 商品名, バリエーションテーマ, バリエーション値, 楽天メイン画像1, 楽天サブ画像1〜8。CHECKBOX_HEADER_NAME（出品CK）。
- **getMVal / ▼マスタ**: 楽天キャッチ(60-80字), ブランド, シリーズ, 原材料(食品), 賞味期限(食品), 保存方法(食品)。その他 COMPARE_ITEMS は masterColMap[`▼マスタ(${項目名})`] で参照。
- **数値列（NUMERIC_COL_NAMES）**: 商品の販売価格, 在庫数, 総個数, 卸値(税抜)(税込), 梱包・総重量・総容量 等。syncAiDataToMaster で数値として扱う列。

**Yahoo.js**

- マスタは **▼設定(Yahooマッピング)** の 4行目（マスタ参照列名）で列を解決。商品名は headerMap['商品名']。画像は 楽天メイン画像1, 楽天サブ画像1〜8 を参照。価格・在庫はマッピングで指定したマスタ列から取得。

**参照列とモールの対応**

- **価格**: コード.js は「商品の販売価格」1列を参照。Yahoo.js はマッピングで指定した列を参照。実際のスプシがモール別列（例: 楽天の販売価格, Yahooの販売価格）の場合、楽天用・Yahoo用にどの列を参照するかは Phase 2 で決める（共通1列を用意するか、モール別列名を Config／マッピングで指定するか）。
- **在庫数**: 出品時は共通でよいため「在庫数」1列のままでよい。

**参照列一覧（最小必須項目に関係するもの）**

- 商品識別子: 親SKU, 子SKU, ASINコード, JANコード  
- 商品名: 商品名  
- 価格: コード.js は「商品の販売価格」1列。Yahoo はマッピング指定列。モール別列の参照方法は Phase 2 で決定  
- 在庫数: 在庫数（共通1列でOK）  
- 画像: ▼マスタ(参考情報(画像URL))、楽天メイン画像1、楽天サブ画像1〜8  
- 説明・キャッチ: ▼マスタ(商品説明), ▼マスタ(楽天キャッチ(60-80字)), ▼マスタ(Yahooキャッチ(20-30字)) 等

書き込み前に、マスタのヘッダー行に上記「列名」が存在するかをチェックし、不足があればアラートで列名を表示して処理を止める。**優先4**ではメニュー「◎行にセット数候補・価格提案をマスタ反映（優先4）」で、マスタの「評価」列に◎が含まれる行を対象に、ASIN貼り付けシートのブロック順で競合セット数・穴のセット数提案を書き込み、続けて既存の価格提案ロジックで販売価格を反映する。  

**セット構成提案**（メニュー「セット構成提案」）では、**AI情報取得data** の記載がある2行目以降の全行を対象に、**ASIN貼り付け（Keepa用）** のブロックから競合セット数・穴のセット数提案を取得し、**▼商品マスタ(人間作業用)** の「最終データ行」の次から親行・子行を追記する。最終データ行は **G列～Z列のいずれかに値がある最後の行** で判定する。親行1行には AI情報取得data の メーカー名・仕入判断・JANコード・商品名・卸値(税抜)・卸値(税込)・JANコードorオリジナルカタログ商品名・ASINコード を書き込み、続く子行には同じ8項目に加え **A.セット商品数** に「〇個で1セット」を1セット数＝1行で複数行書き込む。セット数は競合セット数と穴のセット数提案をマージした重複なし昇順の一覧を使用する。トリガーは人間がメニュー「AI出品ツール → 商品リサーチ → ②出品用 → セット構成提案」を実行すること。

**セット構成提案 — 実装確定仕様（2026-03）**

| 項目 | 確定内容 |
|------|----------|
| **書き込み方式** | 全列上書きではなく**書き込み対象列のみ列単位で setValues()**。書き込む列は以下の11列のみ: メーカー名 / 仕入判断 / JANコード / 商品名ベース / 卸値(税抜) / 卸値(税込) / JANコードorオリジナルカタログ商品名 / 競合店ASINコード / 競合AmazonページURL / 競合価格amazon / A.セット商品数。これ以外の列（計算式が入った列等）は一切変更しない。 |
| **テンプレート行からの計算式コピー** | 新規行の書き込み直前に、スプレッドシートの **1行目**（親SKU用テンプレート）・**2行目**（子SKU用テンプレート）の計算式を `copyTo(PASTE_FORMULA)` でコピー。対象列は **Q列（17列目）〜 HJ列（218列目）**。**amazon価格戦略列および賞味期限列はコピー対象から除外**（セット構成提案の根拠・賞味期限計算値を上書きしないため）。相対参照は書き込み先の行番号に自動調整される。その後に上記11列のデータを上書きするため、書き込み対象列の計算式は最終的にデータ値で置き換えられる。 |
| **親行の競合価格** | 親行の `競合価格amazon` は空欄（子行のみに書き込む） |
| **業務用セット数の計算基準** | Gemini に「1日何包使うか（daily_packets）」を問い合わせ、`ceil(dailyPackets × 360 / packetsPerBag)` で**12ヶ月分**の袋数を業務用セット数とする。`packetsPerBag` は商品名から `NP` パターンで抽出（例: 4P→4, 48P→48）。 |
| **業務用セット数の fallback** | Gemini 失敗時は `compSetMax × 2`（フィルター済み競合最大値の2倍）を使用。`Math.max(bizSet, fallback)` の比較は行わない（Gemini 結果を優先し、bizSet ≤ currentMax の場合のみ引き上げ）。 |
| **12ヶ月上限フィルター** | `sixMonthMaxSets = 360`（12ヶ月相当）を超える競合セット数は **setCounts 構築前に除外**（非◎行の非現実的な大セット数対策）。bizSet もこの上限でキャップ。 |
| **bizSet の引き上げ** | bizSet が currentMax（フィルター済み競合最大値）以下の場合、`min(currentMax × 2, sixMonthCap)` に引き上げ。 |
| **比例配分ギャップ埋め** | 競合セット数はすべて含め、競合＋bizSet をアンカーとして中点分割等でギャップを埋める。最大**15種類**にキャップ。 |
| **ASIN・URL の紐付け** | 子行ごとに、そのセット数の `◎` 行の中で最安値となる競合の ASIN・URL を `競合店ASINコード` / `競合AmazonページURL` に書き込む。競合データのないセット数（ギャップ埋め分等）はこれらを空欄にする。 |
| **セット数ポートフォリオ（Gemini）** | GEMINI_API_KEY が設定されている場合、**Gemini にセット数ポートフォリオを提案**させ、有効な返答（3〜20件・12ヶ月上限内・採用後は最大15種類）ならそれを採用。失敗時は従来ロジック（競合＋業者用＋比例配分）にフォールバック。プロンプトは `docs/SET_COUNT_PROPOSAL_PROMPT.md` を参照。 |
| **AI情報取得data とマスタの対応** | カテゴリ・商品仕様等を参照する際は、**▼商品マスタ(人間作業用)の「商品名ベース」と AI情報取得data の「商品名」または「商品名ベース」が一致した行**の情報を用いる。AI情報取得data に列が無い・空の場合はマスタの同名列（または「▼マスタ(〇〇)」列）を参照する。 |
| **賞味期限** | AI情報取得data の「賞味期限(食品)」を入力日（実行日）から起算して計算。「賞味期限（ある場合のみ）分からない場合は今日で入力」列を優先して書き込む。食品以外は未入力。 |
| **オリジナルカタログ商品名・購入日** | オリジナルカタログ商品名は親＝1行目の値、子＝=M$1。購入日＝入力日（実行日）。 |
| **親行の視覚区分** | 親SKU行は水色（#ADD8E6）で塗り分け。 |
| **セット構成提案の根拠・完了後処理** | Gemini の返答全文を子SKU先頭行の「amazon価格戦略」に記載。完了後に **Amazon カテゴリー自動入力**（Gemini）を実行。 |

### 1.3 不足列の追加方針（確定）

- **価格**: モール別価格列を前提とする。既存コード（楽天は「商品の販売価格」固定）との対応は、Phase 2 で次のいずれかを決定する。
  - **(A)** 楽天用・Yahoo用に共通の「商品の販売価格」列を1本用意し、出力前にコピーする等で運用する。
  - **(B)** モール別列（楽天の販売価格, Yahooの販売価格 等）をそのまま使い、コード側で参照列名をモールごとに指定する（Config や ▼設定(Yahooマッピング) で列名を指定）。※楽天ロジックに触れる場合は HANDOVER 方針に従い最小限に。
- **在庫数**: 出品時は共通でよいため、列名「在庫数」1本で足りる。追加・変更不要と確定する。
- **その他**: 商品識別子・商品名・画像・説明・キャッチは既存列で賄えている。

**Phase 2 で行うこと**: モール別価格列の参照方法（上記 A/B）を決定し、必要に応じて列追加または Config／マッピングの整備を行う。

## Phase 2 実施結果（2026-02 実施）

**方針**: モール別価格列の扱いは **(B) 採用**。マスタの価格列は **販売価格amazon**（Amazon用）、**楽天価格設定**（楽天用）、**Yahoo!価格設定**（Yahoo用）でそれぞれ設定している。Phase 3 で楽天用は「楽天価格設定」、Yahoo用は「Yahoo!価格設定」を参照するようコード／▼設定(Yahooマッピング) で指定する。

### 2.1 マスタへの列追加（必要な場合）

- **結果**: 価格列の追加は**不要**（モール別列を採用したため）。追加した列はなし。

### 2.2 調査項目CSVの有無・配置の決定

- **結果**: 調査項目CSVは**使う**。置き場所は **`参考資料：出品自動化スプシデータ`**。フォーマット案は後で決定する（取得項目・マスタ列対応の定義が必要になったタイミングで整備）。

### 2.3 syncAiDataToMaster の動作確認

- **同期対象列（一覧）**: 以下が AI情報取得data → マスタ親行へ同期される ▼マスタ(〇〇) の列。両方のシートに同名の列がある項目のみ同期される。

- **運用メモ**: 同期された ▼マスタ(〇〇) の内容を、人間が「楽天キャッチコピー AIから取得」といった別列で編集したものを**確定値**として出品に使用している。syncAiDataToMaster が更新するのは ▼マスタ(〇〇) 列のみ。確定値列は同期対象外。
- **確認手順**（あなたがスプシ＋GASで実行し、結果を記入する）:
  1. スプレッドシートで、**AI情報取得data** シートと **▼商品マスタ(人間作業用)** シートが存在することを確認する。
  2. マスタの**親行**（子SKU が空の行）のうち、ASINコード・JANコード・商品名のいずれかで **AI情報取得data** の行と一致するものを1件以上選ぶ。
  3. 上記「同期対象列」のいずれか（例: ▼マスタ(商品説明)、▼マスタ(楽天キャッチ(60-80字))）について、AI情報取得data のその列の値が同期後にマスタの同じ列の親行にコピーされることを確認する。

**今後の方針**: マスタ同期（syncAiDataToMaster）は自動化・効率化の対象とする。要件定義は**出品情報の整理**をするタスクのときにあわせて行う。
