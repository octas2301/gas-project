# モール横断セット数判定 テスト検査結果

## テスト範囲

- **メニュー**: 「モール横断 セット数統合判定テスト」（`menuTestCrossMallSetCountJudge`）
- **入力**: マスタで1行選択 → その行の JAN をキーに実行
- **出力**: シート「モール横断セット数判定」に 23 列 + 基準単品原価
- **範囲外**: マスタへの書き戻し、他メニュー・他シート、複数JAN一括、本番反映

## 検査項目と結果

### 1. 統合セット数・統合根拠

| 項目     | 結果 |
|----------|------|
| 判定精度 | **合格**（目視確認で精度が高い） |
| 備考     | セット数・根拠は実用上問題なし |

### 2. 価格・送料まわり（お客様支払い想定額）

| 項目           | 結果 |
|----------------|------|
| 確認用列       | **価格+送料**（支払い想定額）、**仮定送料**（0＝送料無料扱い、非0＝送料加算） |
| 仕様           | 競合の送料が分からないモールは自社送料を想定送料として加算。**送料無料は加算しない**（送料フラグが付いているもののみ楽天・Yahoo!で自社送料を加算）。 |
| 現状の実装     | 楽天・Yahoo! の**全候補**に自社送料を加算している。`postageFlag`（送料あり=1）を参照していない。 |
| 検査結果       | **要改善**。送料無料の商品（例: 「メール便送料無料」）でも 仮定送料=自社送料・価格+送料=価格+185 となっており、仕様と不整合。 |

## 送料の扱い（API仕様・本番実装後）

### Amazon（Keepa）

- **現状**: 送料込み価格は取得していない。Amazon と API 連携の都合上、モール横断では仮定送料 0 のままでよい。
- **方針**: 将来、送料が取得できるようになったら再検討する。コード変更は不要で、この認識をドキュメントで共有する。

### Yahoo! ショッピング API v3（商品検索）

- **レスポンス**: 各 hit に `hits/shipping` オブジェクトあり。
  - `hits/shipping/code` (integer): **1** = 設定無し、**2** = 送料無料、**3** = 条件付き送料無料
  - `hits/shipping/name` (string): コードに紐づく名前
- **実装**: `shippingCode === 2` のときのみ仮定送料 0。それ以外（1, 3）は自社送料を加算する。

### 楽天 Ichiba API

- **レスポンス**: 各 item に `postageFlag`。**1** = 送料あり、それ以外 = 送料無料扱い。
- **実装**: `postageFlag === 1` のときのみ自社送料を加算し、それ以外は仮定送料 0。
