## 2. Yahoo! 出品（Ver 3.1）確定仕様

### 2.1 商品構造（SKU爆発 × グルーピング）

- マスタの **子SKU 1つ = Yahoo! 上の 1商品（個別ページ）**として展開
- 各商品に共通の `grouping-id`（= マスタの親SKU）を設定し、ページ内でバリエーション切替を可能にする
- **出品対象**: Yahoo! では **子SKUの行の出品CK（レ点）のみ**で判断。親のレ点は楽天用であり、Yahoo!には「親だけレ点で全子を出品」する仕様はない（詳細は 5.2.1）

### 2.2 商品コード（code）生成ルール（画像仕様対策）

- 形式: `親SKU + "-" + 子SKUの末尾`
- 区切り文字は **アンダースコア `_` ではなくハイフン `-`**
  - 理由: Yahoo! 画像仕様で `_` が「商品コードと画像連番の区切り」とみなされ、英字含むSKUで `im-02004` を誘発しうるため

### 2.3 画像連携（API × CSV）

- 送信: Yahoo! Image API `uploadItemImage` をGASから即時アップロード
- ファイル名規約:
  - 1枚目: `商品コード.jpg`
  - 2枚目以降: `商品コード_1.jpg` ...（連番）
- CSV側（`item-image-urls`）: **空欄**で出力（API送信済み画像を、ファイル名一致で自動紐付けさせる）
- 画像取得ロジック:
  - その子SKUの「楽天メイン画像」
  - その子SKUの「楽天サブ画像」（無ければ親SKUの楽天サブ画像を継承）
  - ※「兄弟SKUのメイン画像を集める」処理は重複に見えるため削除済み

## 3. シート構成とマッピング制御（Yahooマッピング）

対象: `▼設定(Yahooマッピング)` シート

- **1行目**: Yahoo! フィールド名（`code`, `name`, `price` ...）
- **2行目**: ファイル種別（`data`, `quantity`, `option`）
  - `data_add.csv` 生成時は `quantity` / `option` 列を自動除外
- **4行目**: マスタ参照列名（入力があればマスタから値を取得）
- **5行目**: 固定値 / 特殊タグ
  - `[[固定値:XX]]`: `XX` をそのまま出力
  - `[[JOIN_BULLETS]]`: マスタ「商品説明の箇条書き①～⑤」を `<br>` で連結
  - `[[JOIN_BULLETS_DIAMOND]]`: 上記に「◆ 」を各項目の先頭に付けて連結（競合のような装飾向け）
  - `[[HTML_CLEAN]]`: HTMLタグ除去
- **6行目**: 出力フラグ（`TRUE/FALSE`）

## 4. 解決済みの技術課題（再発防止メモ）

### 4.1 画像APIの `seller_id` の渡し方

- `uploadItemImage`: `seller_id` は **URLクエリ**で付与（`?seller_id=...`）
  - ボディに入れると `400 Bad Request` を誘発（検証済み）

### 4.2 画像APIのパラメータ名

- 画像データのキーは `image_file` ではなく **`file`**

### 4.3 CSVのフィールド重複（dataとquantityの混入）

- `data_add.csv` 生成時に `quantity_add.csv` 用列が混ざる問題は、**2行目（ファイル種別）で除外**するロジックで解決済み

### 4.4 グルーピングエラー `U-001-0349`

- スペックIDなしの「自由入力モード」を採用
  - `variation1-free-title`（項目名）と `variation1-name`（選択肢名）を出力
  - `variation1-spec-id` は空欄で出力

### 4.5 Yahoo! 認証エラー時のリトライ（2026-02 実装）

**経緯**: 一括出品で約9分・17件送信中に13件目で失敗。API が `Bad Request: Authentication parameters in your request incompleted.` を返した。トークン有効期限や当該リクエストのみの認証不備が疑われた。

**対応内容（Yahoo.js）**:
- **対象**: `runYahooExport` 内「5. APIによる商品登録 & 在庫設定」ブロック（364〜433 行付近）。
- **変更1**: `const itemClient` → `let itemClient`（リトライ時に再代入するため）。
- **変更2**: 認証エラー判定 `isAuthError(msg)` を追加。メッセージが `Authentication ... incompleted` 系か正規表現で判定。
- **変更3**: 1商品分の処理を `runOneProduct(client, product)` に集約（updateItem → setStock → submitItem。it-07004 のときは成功扱いで return）。
- **変更4**: `products.forEach` の `catch (e)` で `isAuthError(e.message)` のときのみ、`getYahooAccessToken(mapSheet)` でトークン再取得 → `itemClient` を新トークンで再生成 → 当該商品だけ `runOneProduct` を再実行。成功なら successCount に含め、再失敗時のみ errorItems に追加。

**未対応（今後の改善候補）**:
- エラーメッセージ: 「使用できないアドレス」ではなく、API の `<Message>` をそのままログ・メールに出すと原因が分かりやすい。
- 一括出品ログ: `[一括出品ログ] 記録失敗: データの行数が範囲の行数と一致しません。` → **対応済み**（コード.js の `logBatchExportProgress`）。`getRange(nextRow, 1, nextRow, 2)` の第3・4引数は「行数・列数」のため、第3引数を `1` に変更（1行2列の範囲に修正）。

### 4.6 発送日情報（「発送日情報なし」の解消）

商品ページで「発送日情報なし」と出る場合、editItem の **発送日情報管理番号** が未設定です。

**手順:**

1. **ストアクリエイターPro で発送日情報を登録**
   - 管理画面: **商品・画像・在庫** → **発送日情報**（または配送関連設定）
   - 「発送日情報を追加」で表示したい文言を登録（例: 「在庫ありの場合、翌営業日発送」）
   - 登録後に表示される **管理番号**（1, 2, 3 …）をメモする

2. **マッピングシートに列を追加**
   - `▼設定(Yahooマッピング)` の **1行目（Yahoo項目名）** に次の2列を追加:
     - `lead-time-instock` … 在庫あり時に表示する発送日文言の管理番号
     - `lead-time-outstock` … 在庫切れ時に表示する発送日文言の管理番号
   - **2行目**: ファイル種別を `data`
   - **5行目**: 固定値で管理番号を指定（例: `[[固定値:1]]`）
   - **6行目**: 出力フラグを `TRUE`

同一の管理番号を両方にしても、在庫あり用・在庫切れ用で別番号を指定してもよいです。未設定のままなら API は送らず、ストア側で「発送日情報なし」のままになります。

**「説明通りなのに発送日情報なしのまま」の場合の確認ポイント**
- **1行目（Yahoo項目名）は完全一致**：「lead-time-instock」「lead-time-outstock」と入力（`lead-time-outstoc` のように k が欠けると送信されません）。※コード側で `lead-time-outstoc` と書かれている列は `lead-time-outstock` として送る互換を入れています。
- **項目名が日本語でも可**：1行目が「在庫あり納期」「在庫なし納期」や「発送日～在庫あり」などでも、Yahoo.js の `_toApiParamName` で `lead_time_instock` / `lead_time_outstock` に変換して送る互換を入れています（2026-02 追加）。ログの「発送日関連とみなしたマッピングの1行目キー」で実際のキー名を確認できます。
- **2行目**: ファイル種別を `data`、**6行目**: 出力を `TRUE` にすること。
- **再送信**: 列を追加しただけでは既存商品は更新されません。**一括出品で該当商品を再登録**するか、該当商品だけ editItem を再度実行すると、発送日情報が反映されます。
- 実行ログの `[DEBUG] 発送日情報: lead_time_instock=..., lead_time_outstock=...` で、想定どおり送られているか確認できます。

### 4.8 商品説明の充実と装飾（競合のような見せ方）

**Yahoo! の仕様（アップロードフォーマット）**
- **caption（商品説明）**: 全角5,000文字以内・**HTML可**（`<b>`, `<br>` など使用可）
- **explanation（商品情報）**: 全角500文字以内・**HTML不可**（◆ ● 【】などの文字はそのまま使える）
- **headline（キャッチコピー）**: 全角30文字以内・HTML不可

**できること**
1. **マスタの列に直接装飾を書く**  
   マッピングで caption / explanation に紐づくマスタ列（例: 商品説明・商品情報）に、◆ ● 【】「」や、caption の場合は `<b>【見出し】</b><br>◆ ポイント1` のように HTML を書けばそのまま反映されます。
2. **箇条書きを ◆ 付きで出す**  
   マッピングの caption または explanation の **5行目** に `[[JOIN_BULLETS_DIAMOND]]` を指定すると、「商品説明の箇条書き①～⑤」が「◆ 項目1」「◆ 項目2」… のように `<br>` で連結されます（競合のような装飾向け）。

**運用のコツ**
- 競合のように「【見出し】＋◆ で箇条書き」にしたい場合は、caption にマスタ列を参照させ、その列に `【蒸し生姜の圧倒的温め力】<br>◆ 国産生姜を丸ごと…` のように HTML と記号を入れるか、5行目で `[[JOIN_BULLETS_DIAMOND]]` を使いマスタの「商品説明の箇条書き①～⑤」に文言を書く。

### 4.7 グルーピング（できている商品とできていない商品がある場合）

**原因として考えられる点**
- **grouping_id の文字種**：API は「半角英数字・ハイフンのみ」有効。親SKUに全角や記号が含まれると、同一グループでも別扱いになる場合があります。
- **マスタの行順**：**親行の直後に、その親の子行が並んでいること**が必要です。親より前に子だけの行があると、その子はどのグループにも入らず、グルーピング対象になりません。
- **親SKUの表記ゆれ**：同じグループにしたい商品の「親SKU」が、行ごとに微妙に違う（スペース・全角半角など）と、別グループとして登録されます。

**コード側の対策（Yahoo.js）**
- editItem 送信時に `grouping_id` を半角英数字・ハイフンにサニタイズして送信するようにしました。親SKUの全角数字・英字は半角に変換してからサニタイズします。
- マッピングで「grouping-id」列に固定値やマスタ列を入れている場合は、同じグループにしたい商品は**同じ値**になるようにしてください。

**確認方法**
- 同一グループにしたい商品の「親SKU」がマスタ上で完全に同一か確認する。
- 一括出品後、ストア反映（submitItem）まで行ったうえで、商品詳細ページでバリエーション切替ができるか確認する。

## 5. EC完全自動出品（Ver 5.0 / Final）要件の要点

### 5.1 目的

- 「出品・在庫設定・画像登録・反映確認」を **ワンクリックで完全自動化**

### 5.2 マスタシート（UI/ステータス領域）

- `▼商品マスタ(人間作業用)` の **HY列以降**をシステム管理領域として使用
- ステータス列（ヘッダー名で列位置を検索）:
  - `楽天CSV`: `未 / 済`
  - `楽天FTP`: `未 / 送信済 / 反映済 / Error`
  - `Yahoo画像`: `未 / 済 / Error`
  - `Yahoo出品`: `未 / 済 / Error`
- 特殊タグ:
  - `[[固定値:XX]]` / `[[JOIN_BULLETS]]` / `[[AUTO]]`（今回は空欄扱い）

### 5.2.1 出品CK（レ点）の要件（楽天・Yahoo! 同時出品）

本プロジェクトは **楽天とYahoo!を同時に出品していく** 前提のため、次のように整理する。

| 項目 | 内容 |
|------|------|
| **親SKUのレ点** | **楽天では必須**。親にレ点を付ける運用を維持する。親のレ点の有無は **Yahoo!の出品判定には使わない**。 |
| **Yahoo!の出品対象** | **子SKUの行のレ点（出品CK）のみ**で判断する。レ点が付いた子SKUの行だけがYahoo!に出品される。 |
| **禁止する仕様** | 「親だけレ点 → Yahoo!にその親に紐づく全子SKUを出品する」という動きは **行わない**。 |
| **理由** | 数量セットごとに利益が異なるため、利益が出ないセット（子SKU）は出品しない。子SKU単位でレ点を付けて出品する。 |

**まとめ**

- 楽天: 親SKUのレ点は従来どおり（必要に応じて使用）。
- Yahoo!: **子SKUのレ点のみ**で出品するかどうかを決める。親のレ点があっても、子にレ点がなければYahoo!には出品しない。

### 5.3 Yahoo!（ItemAPI連携）

- 構造: 子SKUごとに独立商品として登録し、`grouping_id` で束ねる（SKU爆発型）
- API:
  - 画像: `uploadItemImage`（Multipart）
  - 商品: `editItem`（Form-urlencoded）
  - 在庫: `setStock`（Form-urlencoded）
  - **ストアクリエイターpro 反映**: `submitItem` — **既存商品の編集時のみ**APIで自動実行。新規出品の反映はAPI非対応のため手動（下記 5.3.1）
- **seller_id の場所（重要）**
  - 画像API: URLクエリ（`?seller_id=...`）
  - 商品/在庫API: **ボディ（payload）**（`payload["seller_id"] = ...`）
  - 取り違えると `px-04001` になりうる（検証済み）
- パラメータ名（例）:
  - 商品名: `item_name` ではなく `name`
  - 価格: `item_price` ではなく `price`
- 制約/型:
  - `name`: 全角75文字以内（超過時は単語区切りで末尾カットするロジックが必要）
  - `taxable`: 文字列 `"1"` ではなく **数値 `1/0`**
  - `ship_weight`: 空なら `0` を送らず、**パラメータ自体を送らない**
- 直近課題:
  - `taxable` が `[[固定値:1]]` のまま送られてエラーになった
  - `[[固定値:XX]]` / `[[JOIN_BULLETS]]` 等のタグ処理を、値取得メソッド（例: `_getValue`）内で完結させる必要

#### 5.3.1 ストアクリエイターpro 反映（要件定義・実装）

**要件（確定）**

- **自動で行うこと**: 商品登録（editItem）・在庫設定（setStock）まで。**既存商品を編集した場合**は、submitItem で反映まで自動実行する。
- **自動でできないこと**: **新規出品**の反映。Yahoo API 仕様により、新規ページは個別反映API（submitItem）が使えず、**ストアクリエイターpro の「反映」ボタンを手動で押す**必要がある。
- **新規時の運用**: せめて**メールをスマホから開き、そこから手動で反映処理できるようにする**。完了メールに「反映用リンク」と「スマホでの手順」を記載し、タップしてストア管理に近いところまで開けるようにする。

**実装**

- 画像送信 → editItem → setStock → **submitItem**（既存のときは成功、新規のときは it-07004 でスキップ）→ マスタに Yahoo出品済ID 記録 → **完了メール**（下記）。
- 完了メール本文に **【新規出品の反映（スマホから）】** を追加: ストアクリエイターpro の反映画面に近いURL（ストアURLまたはマッピングシートで指定した「反映用URL」）と、「スマホでメールを開く → リンクをタップ → ログイン → 商品ページ → 「反映」ボタン」の手順を記載。
- it-07004 のときはエラーにせず成功扱いとし、ログに「新規のためAPI反映は未対応。ストアクリエイターproで「反映」ボタンを押してください」と残す。

**実行ログ**: シート「▼ログ(システム用)」で、editItem → setStock → submitItem の順と、it-07004 時の案内を確認できる。

**完了メールの反映案内**: メールに載せるリンクは**ストアクリエイターpro トップ**（`https://pro.store.yahoo.co.jp/pro.{sellerId}`）。このURLをタップ → ログイン → 商品ページ → 「反映」で一括反映。B18 に **http で始まるURL** を入れるとそのURLを表示（B18 が「YahooカテゴリID」などURLでない場合はデフォルトの pro.store リンクが使われる）。**2段階認証**は Yahoo 側のためアプリからスキップ不可。

#### 5.5.3 削除の仕様（Yahoo!と同様の考え方）

**A. スプレッドシートでまとめて削除（Yahooの「削除CK」と同様）**

1. マスタで削除したい行に **楽天削除CK** にレ点を付ける。
2. メニュー「楽天」→「楽天 商品を削除...」を実行。
3. 「楽天削除CKの付いた N 件を削除します。よろしいですか？」と確認（ui.alert YES/NO）。
4. YES の場合:
   - 対象行から **商品管理番号**（その行の 親SKU+子SKU から生成、または「楽天商品管理番号」列があればその値）を収集。
   - **item-delete.csv** を生成（Shift_JIS、楽天仕様）。
   - CSV を Google Drive に保存し、**MAKE の削除用 Webhook** に fileId を送信。
5. MAKE 側: 受け取った fileId で Drive から item-delete.csv を取得し、**楽天FTP に item-delete.csv としてアップロード**。
6. 削除実行後、対象行の **楽天削除フラグ** を TRUE、**楽天削除日時** を記録し、**楽天削除CK** のレ点を自動で外す。

**B. MAKE 経由で削除**

- 上記 A の 4〜5 が「MAKE 経由」: GAS は item-delete.csv を Drive に保存し、MAKE に「削除用」であることを示すペイロード（例: `action: "delete", fileId, fileName: "item-delete.csv"`）で Webhook 送信。
- MAKE 側に **削除専用シナリオ** を用意するか、既存の「楽天FTPアップロード」シナリオで **fileName が item-delete.csv のときは別フォルダ／別ファイル名でFTPに置く** など、楽天が受け付ける削除用ファイル名・配置に合わせる。
- 楽天の仕様: 削除用は **item-delete.csv** というファイル名でFTPにアップロードする運用が一般的。

**C. メールから削除（Yahoo の「削除用URL」と同様）**

1. 楽天出品完了メールに、各商品ごとに **削除用URL** を1行記載する。

#### 5.5.5 MAKE側で必要な設定の確認方法

GAS は楽天削除時に **同じ Webhook**（`https://hook.eu1.make.com/...`）に次の JSON を送っています。

| キー | 値 | 説明 |
|------|-----|------|
| `action` | `"delete"` | 削除用であることを示す |
| `fileId` | （Drive のファイルID） | item-delete.csv の ID |
| `fileName` | `"item-delete.csv"` | ファイル名 |
| `source` | `"googleDrive"` | 取得元 |

**確認手順（MAKE 上で行うこと）**

1. **Webhook を受けているシナリオを開く**  
   楽天FTPアップロード用のシナリオ（例: RakutenFTPUpload）を開く。

2. **最初のモジュール「Webhooks – Custom webhook」をクリック**  
   - 「Show only first run」をオフにして、**履歴（Execution history）** を開く。
   - スプレッドシートで「楽天 商品を削除...」を実行した直後なら、**直近の実行**に `action: "delete"` のデータが届いているか確認する。
   - 届いていれば、その実行を開き、**出力データ**に `action`, `fileId`, `fileName` が含まれているか確認する。

3. **分岐の有無を確認**  
   - Webhook の**次**に **Router** や **Filter** があるか確認する。
   - **ある場合**:  
     - 条件が「`action` が `delete`」または「`fileName` が item-delete.csv」になっているか確認。  
     - その先に「Google Drive – Download a file」で `fileId` を使い、**楽天FTP の削除用フォルダ／ファイル名（例: ritem/batch の item-delete.csv）** にアップロードするフローがあるか確認。
   - **ない場合**:  
     - 現状は「通常の出品CSV」だけをFTPに送っている可能性が高い。  
     - **Router を追加**し、`action = "delete"`（または `fileName = "item-delete.csv"`）のときだけ「Drive から item-delete.csv を取得 → 楽天FTP の削除用パスにアップロード」するルートを追加する必要がある。

4. **楽天FTPの削除仕様を確認**  
   - 楽天のマニュアルまたは現在の手動運用で、**削除用CSVのファイル名と配置場所**（例: `item-delete.csv` を特定フォルダに置く）を確認する。  
   - MAKE の FTP モジュールで、削除用ルートではその**ファイル名・パス**にアップロードするように設定する。

**まとめ**

- **「MAKEに流れる口」**: GAS は同じ Webhook に `action: "delete"` 付きで送っているので、**MAKE には届いている**。
- **「削除が実行されない」**場合は、MAKE 側で `action` または `fileName` による**分岐**と、**削除用のFTPアップロード**が設定されているかを上記の手順で確認する。

## 10. 新プロジェクト: 商品情報取得とマスタ効率化

出品前の「商品情報の取得」と「マスターシートの効率化」を新規プロジェクトとして要件定義済み。詳細は Cursor のプラン「商品情報取得とマスタ効率化」または `.cursor/plans/商品情報取得とマスタ効率化_*.plan.md` を参照。

### 10.1 現状フロー（共有用）

- **入力**: 「AI情報取得data」にメーカー名・JAN・商品名・ASIN・参考情報(画像URL) 等を**手入力**。参考情報(画像URL) は対象品一致判定・評価に利用する（RESEARCH_AND_ESTIMATE.md の画像一致・参考画像セット数対策の要件定義を参照）。
- **AI生成**: `generateListingDataComparison` で楽天ジャンル・Yahooカテゴリ/ブランドのCSV検索 ＋ Gemini/GPT で COMPARE_ITEMS 生成。正解列を人間が編集。
- **マスタ同期**: `syncAiDataToMaster` で AI情報取得data の正解を ASIN/JAN/商品名で突合し、**▼商品マスタ(人間作業用)** の対応列へ反映。更新対象はマスタの**親行のみ**（子SKUが空の行）。
- **マスタ**: 親SKU・子SKU・出品CK を軸に楽天・Yahoo 両方のソース。Yahoo は Yahoo.js の `YahooDataBuilder` がマスタ＋「▼設定(Yahooマッピング)」を参照。
- **出品**: マスタの「出品CK=TRUE」の行が楽天CSV出力・Yahoo一括出品の対象。

**ボトルネック**: 商品情報の最初の入力が手作業。マスタ列と参照データ（送料・手数料・サイズ等）の利用がコードに分散している可能性。

### 10.2 商品出品情報フォルダ

- **パス**: プロジェクト内 `商品出品情報/`
- **既存CSV**: `自動出品【通常260221】●月●日（●様）セラセン商品登録一括アップロード(Flat.File.ListingLoader.jp）.csv`  
  - FBA納品・充填率・包材費・自己配送料金・FBA手数料・販売手数料・ダンボールサイズ・地域別送料・楽天/Yahoo手数料率など。
- **方針**: 調査項目をCSVで同フォルダに格納し、GAS から「商品出品情報フォルダ内の指定CSVを読む」共通仕様にする。項目追加・取得元変更はCSV編集で対応し、コード変更を最小にする。

### 10.3 新プロジェクトの要件定義（案）

| 領域 | 要件（案） | 目的 |
|------|------------|------|
| **商品情報の取得** | 仕入元・セラセン・カタログ・既存CSV等から可能な範囲で自動取得し、AI情報取得data またはマスタに流し込む | 手入力列・行を減らす（JAN/ASIN/商品名/画像URL等の自動投入） |
| **取得項目の定義** | 「調査項目CSV」で定義し、GAS はその定義を読んで実行 | 項目追加・取得元変更をCSVで対応 |
| **マスタの効率化** | マスタ列と取得項目の対応を明確化。送料・手数料・サイズ等は商品出品情報のCSVを参照して補完・計算 | 入力ミス削減・計算項目の自動算出 |
| **重複・バリデーション** | 同一JAN/ASIN/子SKUの重複検知、必須項目の欠損チェック | 手動突き合わせの削減 |
| **既存フローとの接続** | AI情報取得data → マスタ同期 → 楽天/Yahoo出品は維持。新機能は「取得」「マスタ補完」の前段として挿入 | 既存自動出品への影響を最小限に |

**非機能**: GAS はフローの塊ごとにファイルを分け、単体で修正・テストしやすくし、他ファイルへの影響を最小限にする。

### 10.4 GAS ファイル分割案（フロー塊ごと）

| ファイル（案） | 役割 | 依存 |
|----------------|------|------|
| **Config.js** | シート名・列名・フォルダID・COMPARE_ITEMS 等の定数一元化 | 他から参照のみ |
| **MasterSheet.js** | マスタの読み書き・ヘッダー検出・親子行判定・列マップ・getColumnIndexMap 等 | Config 参照。楽天/Yahoo/取得から利用 |
| **ProductInfoAcquisition.js** | 商品情報取得（調査項目CSV読込・取得元突合・AI情報取得data またはマスタへの書き込み） | Config, MasterSheet。商品出品情報のCSVは Drive から取得想定 |
| **AiListingData.js** | AI情報取得data 初期化・AIデータ生成（Gemini/GPT）・syncAiDataToMaster | Config, MasterSheet |
| **RakutenExport.js** | 楽天CSV出力（generateRakutenCSV 等）・楽天削除フロー | Config, MasterSheet |
| **Yahoo.js** | Yahoo 専用（現状のまま。必要に応じ Config/MasterSheet の定数・ヘルパーのみ参照） | 現状どおり |
| **コード.js** | メニュー(onOpen)・一括出品モーダル・各フロー呼び出し・Webアプリ削除API等の「つなぎ」 | 上記モジュールを呼ぶ薄い層 |

影響範囲の目安: 商品情報取得 → ProductInfoAcquisition.js と調査項目CSV。マスタ列 → MasterSheet.js と Config。楽天CSV項目 → RakutenExport.js。

### 10.5 すり合わせ用の確認事項（未確定）

- **調査項目CSVの項目**: 取得項目名・取得元（セラセンCSV列名等）・マスタ列対応・必須/任意 をCSVで定義する想定でよいか。
- **取得元**: セラセンCSV・Drive上のCSV・API等、どれを想定するか。取得元が決まると ProductInfoAcquisition の実装方針が決まる。
- **優先度**: 「取得項目の自動投入」を先にするか、「参照データ補完（送料・手数料・サイズの自動算出）」を先にするか。

次のステップ案: 調査項目CSVのフォーマット案と ProductInfoAcquisition.js の入出力仕様を具体化し、ユーザーとすり合わせて確定する。
