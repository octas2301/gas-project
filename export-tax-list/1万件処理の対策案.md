# 1万件を1週間で処理するための対策案

## 前提の数字

- **目標**: 10,000 PDF / 7日 ≒ **1日あたり約1,430件**
- **現状の無料枠（Gemini 3 Flash）**: RPM 5、RPD 20 など → **1日あたり最大20件**程度
- **結論**: 無料枠の**単一APIキーだけでは1万件/週は不可能**。課金か複数キーか、API呼び出しを減らす工夫のいずれかが必要です。

---

## 案1: 課金でクォータを増やす（Gemini API / Vertex AI）

**内容**  
Google AI Studio または Google Cloud で課金を有効にし、Gemini の利用枠を増やす。

**メリット**
- 今のGASの仕様をほぼそのまま使える（エンドポイント・キーの差し替えのみ）
- 1万件/週を安定して回しやすい
- 実装変更が小さい

**デメリット**
- コストがかかる（利用量に応じて従量課金）

**実装**
- 課金プロジェクトで API キーを発行し、設定シートの「GeminiAPIキー」を差し替えるだけ。

---

## 案2: 複数APIキーをローテーション（無料枠のまま）

**内容**  
複数の Google アカウント／プロジェクトでそれぞれ Gemini API キーを発行し、1件処理するごとにキーを切り替えて使う。各キーが RPD 20 なら、50キーで 1日 1,000 件 ≒ 7日で 7,000 件。

**メリット**
- 課金なしでクォータを実質的に増やせる
- 既存の「1PDFあたり1回API」の流れを活かせる

**デメリット**
- キーを多数用意・管理する必要がある（アカウント/プロジェクトが増える）
- 運用がやや重い

**実装**
- 設定シートに「GeminiAPIキー1」「GeminiAPIキー2」… のように列で持つ、または1セルにカンマ区切りで複数キーを入れ、処理ごとにローテーションして使用する。

---

## 案3: テキスト抽出を「まず別手段」にし、Geminiは補助だけにする

**内容**  
PDF からテキストを取る部分を、**まずは Gemini 以外**で行う。

- **テキスト埋め込みPDF**: 外部の PDF→テキスト API や、Drive の「PDF を Google ドキュメントに変換」などでテキスト取得し、正規表現や簡易パースで 仕向人・申告年月日・申告価格・申告番号 を抜く。
- **画像PDF**: 無料の OCR（例: Google Document AI の無料枠、他のOCR API）でテキスト化してから同様にパース。
- **失敗したり曖昧な場合だけ** Gemini に送って補正する。

**メリット**
- Gemini の呼び出し回数を大きく減らせる可能性がある
- レイアウトが固定に近いなら、パースだけでかなり処理できる

**デメリット**
- 実装が複雑になる
- レイアウトがばらつくとパース精度が落ちる
- Document AI 等も別途クォータ・設定が必要

---

## 案4: 実行時間を「実質フル稼働」に近づける

**内容**  
「5分30秒で止めて1分後に再開」はそのままに、**実行可能な時間帯をできるだけ広げる**（例: 0〜7時、9〜12時、13〜18時、20〜24時など、最大5枠を埋める）。複数AI検証は 1 にして、1PDFあたり1リクエストに統一。

**メリット**
- 実装変更が小さい
- 無料枠の「1日20件」を確実に取りにいける

**デメリット**
- RPD 20 の壁は変わらないため、**単独キーでは1週間で約140件が上限**。1万件には届かず、**案1や案2と組み合わせる前提**になる。

---

## 案5: GAS の外で並列処理（Cloud Run / 自前サーバーなど）

**内容**  
PDF の読み取り・転記の「重い部分」を GAS ではなく、Google Cloud Run・Cloud Functions・自前サーバーなどで実行する。複数ワーカーで同時に API を叩き、結果だけをスプレッドシートに書き戻す。

**メリット**
- 並列度を上げやすく、処理時間を短縮しやすい
- 課金と組み合わせればクォータも取りやすい

**デメリット**
- GAS 以外のインフラ設計・運用が必要
- スプレッドシート連携やトリガーとの役割分担の設計が必要

---

## 案6: モデルを「軽いもの」に統一して 1件あたりの時間短縮

**内容**  
Gemini 2.0 Flash Lite など、**応答が速くトークンも少なめのモデル**を指定する。無料枠の「1日20件」は変わらないが、1件あたりの実行時間は短くなり、5分30秒の枠で処理できる件数が少し増える。

**メリット**
- 設定の「Geminiモデル名」を変えるだけなので実装不要
- 429 やレイテンシが減りやすい

**デメリット**
- RPD 20 の制限はそのままなので、1万件/週を実現するには **案1や案2と組み合わせる必要**がある。

---

## おすすめの組み合わせ

| 目的 | おすすめ |
|------|----------|
| **コストを抑えず確実に1万件/週** | **案1（課金）** |
| **無料のまま可能な限り件数を増やす** | **案2（複数APIキー）** ＋ 案4（時間帯を最大活用）＋ 案6（軽いモデル） |
| **まずは実装コストを抑えたい** | **案1（課金）** でキー差し替えのみ |
| **中長期でAPI呼び出しを減らしたい** | **案3（テキスト抽出を別手段＋Geminiは補助）** を検討 |

---

## 補足: ログに出ている「gemini-2.0-flash」

実行ログの 429 エラーに `model: gemini-2.0-flash` と出ている場合、**いまも gemini-2.0-flash が使われている**可能性があります（無料枠ではクォータ 0 のモデルです）。

- 設定シートの **「Geminiモデル名」** を **空** にするか、**`gemini-3-flash-preview`** にすると、Gemini 3 Flash の枠（RPM 5, RPD 20）が使われます。
- コードのデフォルトは `gemini-3-flash-preview` に変更済みなので、**clasp push が反映されているか** と、**設定シートに「Geminiモデル名」で 2.0-flash が入っていないか** を確認してください。
